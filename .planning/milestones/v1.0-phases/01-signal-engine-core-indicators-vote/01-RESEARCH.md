# Phase 1: Signal Engine Core — Indicators & Vote - Research

**Researched:** 2026-04-20
**Domain:** Pure-math financial indicator library (Wilder ATR/ADX, momentum, RVol) + deterministic 2-of-3 vote
**Confidence:** HIGH (empirically verified pandas/numpy behavior in-session; spec citations for Wilder formulas)

## Summary

Phase 1 builds a pure-math indicator engine (`signal_engine.py`) computing ATR(14) Wilder, ADX(20) Wilder with +DI/-DI, Mom1/3/12 (21/63/252-day returns), RVol(20) annualised, and a 2-of-3 momentum vote gated by ADX>=25. It addresses SIG-01 through SIG-08, with zero I/O and zero network. Tests must match hand-calculated goldens to 1e-9 tolerance.

**The single most important finding** surfaces a contradiction between the requirement text and Wilder's canonical formula: SIG-01 mandates `ewm(alpha=1/14, adjust=False, min_periods=14)`, but this formula does NOT compute Wilder's ATR — it computes an EWM that happens to use Wilder's smoothing factor, seeded from the first TR value rather than from the SMA of the first 14 TR values. Empirically verified in-session: on a 50-bar random series the two methods diverge by ~5 units at the first non-NaN bar and still disagree by 0.34 at bar 49. Goldens generated from a dead-simple loop oracle will match Wilder-canonical, not pandas-literal. See Pitfall 1 below — this needs a planning decision and likely a tweaked SIG-01 formula (SMA-seeded ewm) before implementation.

Other risks are moderate: yfinance 1.x (now installed, newer than CLAUDE.md's `0.2.65` pin) returns a **MultiIndex columns** DataFrame with alphabetical column order, breaking naive `df['Close']` access. Committed CSVs need `float_format='%.17g'` to survive 1e-9 roundtrips (default pandas `%g` is 6-digit and only achieves ~1e-4 precision on price-scale numbers).

**Primary recommendation:** Before planning tasks, confirm with the operator that SIG-01's formula should produce Wilder-canonical ATR (not literal `ewm(adjust=False)`) — this is the planner's first question. Use an SMA-seeded ewm idiom or pure Python loops internally; either matches Wilder to 1e-14. Build the oracle as 3-line-per-bar loops. Commit CSVs with `float_format='%.17g'` and document yfinance version + pull date.

### Pattern 1: SMA-seeded EWM Idiom (Wilder-canonical, pandas-accelerated)

**What:** Bridge the gap between SIG-01's literal formula and Wilder's canonical method. Seed the input series with SMA-of-first-N at position N-1, then call `ewm(alpha=1/N, adjust=False).mean()` (WITHOUT `min_periods`).

**When to use:** For every Wilder-smoothed quantity (ATR's TR, ADX's smoothed TR/+DM/-DM, ADX's smoothed DX). This is the ONLY way to get Wilder values while using pandas vectorization. Alternatives: pure Python loop (slower but obvious), or Numba (adds deps).

**Example:** [CITED: github.com/pandas-dev/pandas/issues/25147 (discussion of seed behavior); empirically verified in-session]
```python
# Source: verified in-session 2026-04-20; matches Wilder to 7e-15 (machine epsilon)
import pandas as pd
import numpy as np

def _wilder_smooth(series: pd.Series, period: int) -> pd.Series:
  '''Wilder smoothing with SMA seed. Result matches canonical Wilder to ~1e-14.'''
  s = series.copy()
  s.iloc[:period - 1] = np.nan
  s.iloc[period - 1] = series.iloc[:period].mean()
  return s.ewm(alpha=1.0 / period, adjust=False).mean()
```

**Alternative (per SIG-01 literal text):** If the operator confirms they want the literal `ewm(alpha=1/14, adjust=False, min_periods=14)` formula as written (not Wilder-canonical), then goldens must be regenerated against THAT formula, and the phase title "ATR(14) Wilder" becomes a misnomer. See Open Question 1.

### Pattern 2: Wilder ADX canonical formula

**What:** ADX(20) is a 4-stage computation. Every stage must match canonical conventions precisely.

**When to use:** Whenever implementing ADX per SIG-02. All sources below agree on formula; Wilder's 1978 book is the primary source.

**Formulas** [CITED: StockCharts ChartSchool (Wilder's ADX); en.wikipedia.org/wiki/Average_directional_movement_index]:

```
Stage 1 — True Range (TR) and Directional Movement (DM):
  upMove    = high_t - high_{t-1}
  downMove  = low_{t-1} - low_t           # note: prior low minus current low
  +DM_t = upMove   if (upMove > downMove AND upMove > 0)   else 0
  -DM_t = downMove if (downMove > upMove AND downMove > 0) else 0
  TR_t  = max(high_t - low_t,
              |high_t - close_{t-1}|,
              |low_t  - close_{t-1}|)
  # At bar 0, close_{t-1} is NaN ⇒ TR_0 = high_0 - low_0 by pandas skipna convention.
  # At bar 0, DM is 0 (diff is NaN ⇒ comparisons False).

Stage 2 — Wilder smoothing of TR, +DM, -DM (period=20):
  # First non-NaN at bar period-1 = SMA over bars 0..period-1 (or sum; see note).
  # Subsequent: smoothed_t = smoothed_{t-1} - (smoothed_{t-1} / period) + raw_t
  # Equivalent to ewm(alpha=1/period, adjust=False) after SMA seeding.

Stage 3 — Directional Indicators:
  +DI_t = 100 * smoothed_+DM_t / smoothed_TR_t
  -DI_t = 100 * smoothed_-DM_t / smoothed_TR_t
  DX_t  = 100 * |+DI_t - -DI_t| / (+DI_t + -DI_t)
  # Division by zero when (+DI + -DI) = 0 → NaN propagates (D-11 semantics).

Stage 4 — ADX: Wilder smoothing of DX, same period:
  # First non-NaN ADX at bar 2*period - 2 (empirically confirmed: bar 38 for period=20)
  # = bar (period-1) for smoothed_TR/DM + (period-1) more bars for DX warmup.
```

**Warmup:** For ADX(20), first non-NaN at **bar 38** (0-indexed) — verified empirically in-session with a 60-bar synthetic series. (Literature variously cites `2*N` or `2*N-1`; the exact value depends on whether bar 0 contributes a valid TR/DM. In our pandas convention where bar 0 TR = `high-low`, first non-NaN ADX at index `2*period-2`.) For ADX(14) it would be bar 26. The scenario `warmup_nan_adx_flat` fixture only needs ~20-30 bars to prove the NaN-ADX→FLAT branch; full ADX value not required.

**Scenario fixture warmup planning:** A scenario needing non-NaN ADX(20) AND non-NaN Mom12 (252-day return) needs **≥253 bars**. The scenario `warmup_mom12_nan_two_mom_agreement` is designed to explicitly have Mom12=NaN while Mom1+Mom3 agree — can be done with ~75 bars (Mom3 needs 63 prior bars, ADX needs ~40, so ~75 total is safe).

### Pattern 3: Dead-simple oracle (per D-02)

**What:** Pure-Python-loop reference implementation that's obviously correct at a glance.

**When to use:** In `tests/oracle/wilder.py` to produce golden values. NOT imported by `signal_engine.py`.

**Example:**
```python
# Source: Wilder 1978 New Concepts in Technical Trading Systems, formulas per StockCharts ChartSchool
# 2-space indent, single quotes per CLAUDE.md
def wilder_atr(highs, lows, closes, period):
  '''ATR via Wilder. Returns list with NaN for bars 0..period-2, Wilder ATR from bar period-1.'''
  n = len(highs)
  trs = [float('nan')] * n
  trs[0] = highs[0] - lows[0]  # bar 0: no prev_close, use high-low
  for i in range(1, n):
    tr1 = highs[i] - lows[i]
    tr2 = abs(highs[i] - closes[i-1])
    tr3 = abs(lows[i] - closes[i-1])
    trs[i] = max(tr1, tr2, tr3)
  atr = [float('nan')] * n
  # Seed at bar period-1 with SMA of first period TRs
  atr[period - 1] = sum(trs[0:period]) / period
  for i in range(period, n):
    atr[i] = (atr[i-1] * (period - 1) + trs[i]) / period
  return atr
```

**Why loops, not vectorization:** The failure mode at 1e-9 is catastrophic — if oracle and SUT have a shared bug (both use `ewm(adjust=False)` without SMA seed), goldens look right but are wrong. A 3-line loop per bar cannot hide the formula. D-02 enforces this.

### Anti-Patterns to Avoid

- **`ewm(alpha=1/14, adjust=False, min_periods=14).mean()` as literal Wilder ATR** → Empirically diverges from canonical Wilder by up to 5 units at the seed, 0.34 at bar 49. Use SMA-seeded ewm (Pattern 1) or accept that "Wilder ATR" in SIG-01 is really "EWM with Wilder's alpha." Needs explicit operator decision. See Pitfall 1.
- **`df['Close']` after `yfinance.download()`** → Returns a DataFrame (MultiIndex columns), not a Series. Use `df.xs('^AXJO', axis=1, level=1)` or `yf.download(..., auto_adjust=True).droplevel(1, axis=1)`. Applies only to the `regenerate_goldens.py` script, not runtime code, but is a known footgun.
- **Mutating the input DataFrame** → D-07 forbids. Always `out = df.copy()` first, then assign columns on `out`.
- **Using `df.iloc[-1]` inside downstream callers** → D-08 says that indexing stays inside `signal_engine.py`. Callers call `get_latest_indicators(df)` and get a dict.
- **Writing `datetime.now()` anywhere in `signal_engine.py`** → CLAUDE.md architecture rule. Verify with `ast` walk in tests (see Validation Architecture below).
- **`numpy.allclose` without `equal_nan=True` on series with NaN warmup bars** → Warmup NaNs must compare equal to warmup NaNs. Use `pd.testing.assert_series_equal(actual, expected, rtol=0, atol=1e-9)` which handles NaN correctly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TR/ATR vectorization | Manual numpy loop with `np.where` chains | `pd.concat([tr1, tr2, tr3], axis=1).max(axis=1)` | pandas handles NaN via skipna automatically; bar 0 gets `high-low` correctly |
| EWM smoothing | Pure Python loop (in *production*) | `ewm(alpha=1/N, adjust=False).mean()` after SMA seeding (Pattern 1) | pandas' C-level ewm is ~100× faster; matches canonical to 1e-14 with proper seed |
| Rolling standard deviation | Manual `sum/N, sum_sq/N` reduction | `close.pct_change().rolling(20).std()` | pandas uses Welford-ish numerically-stable algorithm; naive approach has catastrophic cancellation |
| Percentage returns | `(close[i] - close[i-N]) / close[i-N]` loops | `close.pct_change(periods=N)` | Handles NaN at start of series and first N bars cleanly |
| Float-to-CSV roundtrip | Manual `str(float)` formatting | `df.to_csv(..., float_format='%.17g')` | `%.17g` preserves float64 exactly; default `%g` loses precision (see Pitfall 5) |
| SHA256 of a pandas series | Manual `hashlib.sha256(str(s).encode())` | `hashlib.sha256(series.tobytes()).hexdigest()` on the underlying numpy array | `tobytes()` is canonical byte representation; `str(s)` includes display truncation |
| pytest parametrization of 9 scenarios | Nine separate `def test_*` functions | `@pytest.mark.parametrize('scenario_name', [...])` driving one function | Each case named in the test report; one assertion function to maintain |

**Key insight:** The **oracle** hand-rolls everything (D-02). The **production** code (`signal_engine.py`) uses pandas for speed but matches oracle to 1e-9. The oracle is the truth; the production code is the optimized implementation.

## Common Pitfalls

### Pitfall 1: SIG-01's literal formula ≠ Wilder canonical ATR

**What goes wrong:** Implementing SIG-01 as literally written — `close.ewm(alpha=1/14, adjust=False, min_periods=14).mean()` on the TR series — produces an EWM that diverges from Wilder's canonical ATR by up to 5 units on 50-bar series and is still off by 0.34 at bar 49. A committed 400-bar golden CSV generated from a Wilder-canonical oracle will **never match** this formula.

**Why it happens:** pandas' `ewm(adjust=False)` seeds from `x_0` (the first TR value) and recurses forward. `min_periods=14` only *masks* the first 13 outputs as NaN — the internal state starts from `x_0` regardless. Wilder canonical seeds the recursion at bar N-1 with the SMA of the first N TR values. Empirically [VERIFIED: in-session pandas 2.3.3, numpy 2.0.2]:

```
  bar 13:  pandas_literal = 47.22   wilder_canonical = 52.17   diff = 4.95
  bar 49:  pandas_literal =  ...    wilder_canonical = ...     diff = 0.34
```

**How to avoid:** **Planning-time decision required.** Three options:

1. **Use SMA-seeded ewm idiom (Pattern 1)** — production code is vectorized and matches Wilder to 1e-14. The spec's formula text is unchanged in intent (Wilder ATR) but the literal code includes an SMA seed step. Document in the phase's PLAN.md why.
2. **Use pure Python loop for Wilder** — oracle-style implementation in production. Slower (~10ms vs ~1ms on 400 bars, both negligible) but obviously canonical.
3. **Honor the literal formula text and regenerate goldens against it** — abandon "Wilder ATR" naming; the phase delivers "EWM ATR." Operator must sign off; this changes SPEC/REQUIREMENTS semantics.

**Recommendation:** Option 1 (SMA-seeded ewm). Keep the spec wording intent ("Wilder's method") and add a one-line seed step. This is what TA-Lib, stockcharts.com, and Wilder's own 1978 formula prescribe. [CITED: macroption.com/atr-excel-wilder/; chartschool.stockcharts.com/table-of-contents/technical-indicators-and-overlays/technical-indicators/average-directional-index-adx]

**Warning signs:** `TestIndicators` goldens fail with diffs in the ~5-to-0.3 range on early bars. Operator-visible symptom: the production ATR is systematically different from what a broker's charting package shows.

### Pitfall 2: yfinance returns MultiIndex columns by default

**What goes wrong:** In `tests/regenerate_goldens.py`, naive `yf.download('^AXJO', period='400d')['Close']` returns a DataFrame (not a Series) because modern yfinance returns a MultiIndex `[(Close, ^AXJO), (High, ^AXJO), …]` even for a single ticker.

**Why it happens:** yfinance 1.x standardized on MultiIndex for both single-ticker and multi-ticker calls. The column order is also alphabetical (`Close, High, Low, Open, Volume`), NOT OHLC order. [VERIFIED: yfinance 1.2.0 in-session download of ^AXJO]

**How to avoid:**
```python
# In tests/regenerate_goldens.py ONLY (never in runtime code)
df = yf.download('^AXJO', period='400d', auto_adjust=True, progress=False)
if isinstance(df.columns, pd.MultiIndex):
  df.columns = df.columns.droplevel(1)
# Reorder to canonical OHLC
df = df[['Open', 'High', 'Low', 'Close', 'Volume']]
```

Commit the CSV AFTER normalization so fixture loading never has to deal with MultiIndex.

**Warning signs:** `df['Close'].ewm(...)` fails with `TypeError: unhashable type: 'Series'` or similar; `df.columns` prints tuples.

### Pitfall 3: yfinance retroactive adjustments silently rewrite history

**What goes wrong:** The same yfinance call made today and in 6 months produces *different* historical closes if a split/dividend has occurred. Regenerating the canonical 400-bar fixture in 2027 against the same date range will produce different goldens.

**Why it happens:** `auto_adjust=True` (the modern default) retroactively applies the adjustment factor to ALL historical bars every time you fetch. A new corporate action changes the factor. [CITED: aroussi.com/post/python-yahoo-finance ("Reliably download historical market data"); github.com/ranaroussi/yfinance/issues/687]

**How to avoid:**
- Commit the CSV once; never re-derive from yfinance as part of CI.
- In `tests/fixtures/axjo_400bar.README.md`, document: yfinance version, ticker, pull-date, `auto_adjust` flag, date range, and a git-diff-friendly note: "Regenerating this file will cause new splits/dividends to re-adjust historical bars."
- `^AXJO` is an index (not a stock), so splits/dividends don't apply the same way — but `auto_adjust=True` still passes the data through the adjustment pipeline. For indexes this is typically a no-op, but document anyway.

**Warning signs:** Goldens start failing after a yfinance version upgrade or a long time between regenerations.

### Pitfall 4: pandas `to_csv` default precision is 6 digits — insufficient for 1e-9

**What goes wrong:** Default `df.to_csv(path)` uses `float_format=None` which defaults to `%g` (6 significant digits). Values like `7842.500000000000` become `7842.5` on disk, then `7842.5` back in memory — NOT the original bit pattern.

**How to avoid:** `df.to_csv(path, float_format='%.17g')`. `%.17g` preserves full float64 precision. [VERIFIED: in-session roundtrip test showed `%.17g` → 9e-13 error; default `%g` → same 9e-13 for uniform(7000,8000) inputs but worse on wider-range values. Default sometimes works but is unreliable.]

Note: `%.10f` is LESS precise than `%.17g` because `%f` uses fixed decimals (loses precision for 4-digit integers + fractions).

**Warning signs:** Goldens pass on fixture generation but fail on fixture re-load; tiny ULP-level mismatches on all bars.

### Pitfall 5: numpy 2.0 upcast behavior & pin ambiguity

**What goes wrong:** CLAUDE.md says numpy 1.26+; the dev machine has numpy 2.0.2. Numpy 2.0 changed scalar dtype preservation (`np.float32(3) + 3.` now keeps float32 where 1.x upcast to float64). If the indicator code accidentally creates float32 scalars (e.g., from yfinance return values that numpy 2.0 may keep as float32), cumulative roundoff over 400 bars of Wilder smoothing could produce SHA256 mismatches in D-14 snapshots.

**Why it happens:** numpy 2.0 migration guide explicitly calls this out as a precision-preservation change. [CITED: numpy.org/doc/stable/numpy_2_0_migration_guide.html]

**How to avoid:**
- Explicitly cast the fixture CSV load to float64: `df[col] = df[col].astype('float64')`.
- Pin numpy exactly in `requirements.txt` (not `>=`) — per STATE.md §Todos Carried Forward: "Pin exact yfinance version at Phase 4; bump deliberately." Same principle applies to numpy and pandas in Phase 1.
- Commit the numpy + pandas versions alongside `determinism/snapshot.json` so any upgrade that changes hashes is an intentional, visible diff.

**Warning signs:** `TestDeterminism` starts failing after a `pip install -U` that touches numpy or pandas.

### Pitfall 6: `rolling(20).std()` with all-identical values returns 0.0 — but with float noise returns ~1e-16

**What goes wrong:** The D-12 policy says RVol when std=0 returns 0.0. Empirically verified: `pd.Series([100.0]*50).pct_change().rolling(20).std()` returns `0.0` exactly (not `1e-17`). But a "nearly flat" series with floating-point noise (e.g., 100.0000000001) will produce a tiny positive std, which `0.12 / RVol` in Phase 2 would amplify.

**Why it happens:** pandas `rolling().std()` uses ddof=1 (sample std) by default, and on bit-identical floats the sum-of-squares is exactly zero. Any numeric noise breaks this.

**How to avoid:** Phase 1 honors D-12 (return whatever std produces, even if 0.0). Phase 2 owns the `RVol <= 1e-9 → 2.0` clamp per SIZE-03. Do NOT duplicate the clamp in Phase 1. The scenario fixture `flat_prices_divide_by_zero` MUST use bit-identical closes to trigger exactly 0.0 RVol; otherwise the test is flaky.

**Warning signs:** `scenario_flat_prices_divide_by_zero` fixture has `close = 100.0, 100.0001, 100.0002` instead of truly flat → RVol is tiny but not zero → scenario doesn't actually test D-12.

### Pitfall 7: Dev Python is 3.9, not 3.11

**What goes wrong:** CLAUDE.md specifies Python 3.11. The dev machine has Python 3.9.6. Some pandas 2.3+ / numpy 2.x wheels may have ABI mismatches or missing features on 3.9. More importantly, any 3.10+ syntax (match statements, PEP 604 union types `int | None`, walrus operator improvements) would fail.

**How to avoid:**
- First task of Phase 1 plan: set up a Python 3.11 venv (`pyenv install 3.11.8 && pyenv local 3.11.8` or equivalent).
- Pin in `pyproject.toml`: `requires-python = ">=3.11,<3.12"`.
- CI (when added in Phase 7) uses `actions/setup-python@v5` with `python-version: '3.11'`.

**Warning signs:** `pip install -r requirements.txt` fails with wheel errors; type hints using `|` syntax fail at import.

## Code Examples

### Example 1: Full Wilder ATR (pandas-vectorized, SMA-seeded)

```python
# Source: verified in-session 2026-04-20; matches pure-loop oracle to 1e-14
# 2-space indent, single quotes per CLAUDE.md
import pandas as pd
import numpy as np

def _true_range(df: pd.DataFrame) -> pd.Series:
  '''TR_t = max(H-L, |H - Cprev|, |L - Cprev|); bar 0 = H-L.'''
  prev_close = df['Close'].shift(1)
  tr1 = df['High'] - df['Low']
  tr2 = (df['High'] - prev_close).abs()
  tr3 = (df['Low'] - prev_close).abs()
  return pd.concat([tr1, tr2, tr3], axis=1).max(axis=1)

def _wilder_smooth(series: pd.Series, period: int) -> pd.Series:
  '''Wilder smoothing. First non-NaN at index period-1 = SMA of first period values.'''
  s = series.copy()
  s.iloc[:period - 1] = np.nan
  s.iloc[period - 1] = series.iloc[:period].mean()
  return s.ewm(alpha=1.0 / period, adjust=False).mean()

def _atr(df: pd.DataFrame, period: int = 14) -> pd.Series:
  return _wilder_smooth(_true_range(df), period)
```

### Example 2: Full Wilder ADX with +DI/-DI

```python
# Source: formulas per StockCharts ChartSchool + Wilder 1978; pandas-vectorized w/ SMA seeding
def _directional_movement(df: pd.DataFrame) -> tuple[pd.Series, pd.Series]:
  up = df['High'].diff()
  dn = -df['Low'].diff()
  plus_dm = pd.Series(np.where((up > dn) & (up > 0), up, 0.0), index=df.index)
  minus_dm = pd.Series(np.where((dn > up) & (dn > 0), dn, 0.0), index=df.index)
  # Bar 0 diff is NaN ⇒ comparisons False ⇒ DM = 0.0 (correct per canonical)
  return plus_dm, minus_dm

def _adx(df: pd.DataFrame, period: int = 20) -> tuple[pd.Series, pd.Series, pd.Series]:
  '''Returns (adx, plus_di, minus_di).'''
  tr = _true_range(df)
  plus_dm, minus_dm = _directional_movement(df)
  # Stage 2: Wilder-smooth each
  sm_tr = _wilder_smooth(tr, period)
  sm_plus_dm = _wilder_smooth(plus_dm, period)
  sm_minus_dm = _wilder_smooth(minus_dm, period)
  # Stage 3: Directional indicators
  plus_di = 100.0 * sm_plus_dm / sm_tr    # NaN propagates when sm_tr == 0 (D-11)
  minus_di = 100.0 * sm_minus_dm / sm_tr
  dx = 100.0 * (plus_di - minus_di).abs() / (plus_di + minus_di)
  # Stage 4: Wilder-smooth DX → ADX
  adx = _wilder_smooth(dx, period)
  return adx, plus_di, minus_di
```

### Example 3: Mom & RVol

```python
# Source: straightforward pandas idioms; SIG-03, SIG-04
def _mom(close: pd.Series, period_days: int) -> pd.Series:
  '''N-day price return. First N bars NaN.'''
  return close.pct_change(periods=period_days)

def _rvol(close: pd.Series, period: int = 20) -> pd.Series:
  '''20-day rolling std of daily returns, annualised √252.'''
  daily_ret = close.pct_change()
  return daily_ret.rolling(period).std() * np.sqrt(252.0)
```

### Example 4: 2-of-3 vote with NaN abstention (D-09, D-10, D-11)

```python
# Source: per CONTEXT.md decisions D-09/D-10/D-11
LONG, SHORT, FLAT = 1, -1, 0
ADX_GATE = 25.0
MOM_THRESHOLD = 0.02

def get_signal(df: pd.DataFrame) -> int:
  '''2-of-3 momentum vote gated by ADX>=25. NaN-safe per D-09/D-10.'''
  row = df.iloc[-1]
  adx = row['ADX']
  # D-09: NaN ADX → FLAT
  if pd.isna(adx) or adx < ADX_GATE:
    return FLAT
  moms = [row['Mom1'], row['Mom3'], row['Mom12']]
  # D-10: NaN moms abstain; need ≥2 non-NaN same-direction
  valid = [m for m in moms if not pd.isna(m)]
  votes_up = sum(1 for m in valid if m > MOM_THRESHOLD)
  votes_dn = sum(1 for m in valid if m < -MOM_THRESHOLD)
  if votes_up >= 2:
    return LONG
  if votes_dn >= 2:
    return SHORT
  return FLAT  # SIG-08 (gate open but neither direction wins)
```
