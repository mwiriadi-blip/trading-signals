<objective>
Implement the public, unauthenticated `/accept-invite` wizard (RBAC-03 acceptance flow). Multi-step flow per D-04: GET /accept-invite?token=<raw> → step 1 password page → POST validates+hashes password+consumes token+creates user → redirect to TOTP enroll (step 2 reuses existing /enroll-totp) → POST /accept-invite/device → optional trust-device cookie → redirect to dashboard.

Purpose: Phase 34 minted invite tokens; Phase 37 wires the user-facing acceptance flow. The wizard must reject expired/consumed tokens with a dedicated 200 error page (D-07, NOT a redirect to /login), prevent step-skipping via server-validated step cookie, and never log raw tokens.

Three review-consensus fixes are central to this revision:
- **REVIEW #1 (deployment blocker)**: register the new route in `web/app.py::create_app()` BEFORE the AuthMiddleware add (Phase 13 D-06 invariant). Without this, all wizard routes 404 on deploy.
- **REVIEW #2 (partial-enrollment recovery)**: explicit decision recorded — partial-enrollment (User row created, TOTP secret missing) is a documented degraded state in v1.3. The invite token is single-use; a returning invitee with a consumed token sees the standard error page. Admin must issue a new invite for the same email to recover (which itself will fail at consume because the email row exists — full recovery requires a follow-up "delete partial user" admin action, deferred). This plan does NOT auto-delete partial users; it accepts the degraded path and surfaces a regression test for it.
- **REVIEW #6 (TOTP coupling)**: ZERO modifications to `web/routes/totp/__init__.py`. The wizard issues a `tsi_enroll` cookie carrying `next='/accept-invite/device'` in its payload; the existing totp post-enroll handler already reads `payload.get('next', '/')` (line 203) and redirects there. No new TOTP-side hook needed.

Output: New `web/routes/invite/` package; PUBLIC_PATHS extended; `web/app.py` explicitly registers the route before AuthMiddleware; `tests/test_web_invite.py` flips from Wave 0 stubs to RED→GREEN coverage for RBAC-03.
</objective>
