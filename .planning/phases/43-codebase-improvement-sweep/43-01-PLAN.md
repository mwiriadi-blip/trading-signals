<objective>
Fix crash-email data leak (HIGH — F&F blocker, ASVS L1 tenant isolation).

`render_crash_email` currently sends the raw `state` dict which contains all users' trade data and may contain credentials (TOTP secrets, magic-link hashes, emails). Switch from blocklist redaction to a strict ALLOWLIST: only the keys explicitly named below may appear in the rendered body. Unknown / new keys are excluded by default, preventing future leaks when state schema grows.
</objective>

<read_first>
- notifier/templates.py (current render_crash_email implementation)
- tests/test_tenant_isolation.py (the currently-skipped acceptance test)
- state_manager/__init__.py (state schema — to enumerate every top-level key for allowlist decisions)
- auth_store.py (to confirm which keys contain credentials)
- 43-REVIEWS.md (Gemini+Codex HIGH-priority allowlist directive)
</read_first>

<action>
Implement allowlist redaction per consensus review (D-01).

1. In `notifier/templates.py`, define a module-level constant:

   `CRASH_EMAIL_STATE_ALLOWLIST = frozenset({"schema_version", "app_version", "last_run_ts", "markets", "instrument_universe"})`

   (Confirm exact key names by reading state_manager — adjust to actual top-level system-metadata keys. Per-user, credential, and trade keys are NEVER added to this set.)

2. Add `_redact_state_for_crash_email(state: dict) -> dict` that returns a NEW dict containing ONLY keys in `CRASH_EMAIL_STATE_ALLOWLIST`. Apply recursively if any allowlisted value itself is a dict (defence-in-depth — current schema does not require recursion, but the helper must not pass through nested user-scoped substructures).

3. In `render_crash_email`, call `_redact_state_for_crash_email(state)` before serialising. Replace excluded content with the literal string `"[REDACTED]"` at the top-level summary so operators can distinguish "absent" from "hidden" during debugging (Gemini LOW suggestion).

4. Update `tests/test_tenant_isolation.py::test_crash_email_body_redacts_other_users`:
   - Remove the `@pytest.mark.skip` decorator.
   - Seed state with two users, each with unique sentinel values in `trade_log`, `positions`, `pnl`, `email`, `totp_secret`, `magic_link_hash` (e.g. `"SENTINEL_USER_A_TRADELOG"`, `"SENTINEL_USER_B_TOTP"`, etc.).
   - Assert the rendered body does NOT contain any sentinel substring.
   - Assert it DOES contain at least one allowlisted system-metadata value (proves the redaction did not nuke everything).

Do NOT use a blocklist of `["trade_log","positions","pnl"]` — that is the exact anti-pattern both reviewers flagged.
</action>
