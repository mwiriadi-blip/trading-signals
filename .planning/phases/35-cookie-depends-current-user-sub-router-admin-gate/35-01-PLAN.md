<objective>
Add the canonical `get_user_by_email(email, path=None) -> dict | None` helper to `auth_store/_users.py` with case-insensitive email comparison, re-export it via `auth_store/__init__.py`, and lock its behaviour with unit tests including case normalization and duplicate-email handling.

Purpose: Without this helper there is no way to translate `payload['u']` (WEB_AUTH_USERNAME) into the `uid` field stored in `auth.json users[]` — the entire D-01..D-05 cookie-payload extension hinges on it being importable from `auth_store`. Case-insensitive comparison defends against operator-supplied emails being entered with inconsistent case across the login flow vs the bootstrap step.

Output: One new function + one re-export + one new test class. No behaviour change to any caller in this plan.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-01-01 | Information disclosure | get_user_by_email | accept | Caller is auth middleware running server-side; email is already a tenant identifier in auth.json. No PII surfaced beyond what callers already have via get_user(uid). |
| T-35-01-02 | Tampering | get_user_by_email caller | mitigate | Function performs strict equality on lowercased values — no regex, no fnmatch, no SQL. No injection surface. |
| T-35-01-03 | Denial of service | linear scan | accept | users[] is bounded ≤single-digit entries in v1.3 single-admin scope; linear scan O(n) is fine. Re-evaluate at >100 users. |
| T-35-01-04 | Elevation of privilege | role inference | mitigate | get_user_by_email returns the full row but does NOT itself grant access — role checks live in `require_admin` (Plan 03). This helper is a pure lookup. |
| T-35-01-05 | Spoofing | case-insensitive equality enabling lookup collision | accept | Case-insensitive equality is the standard contract for email; Phase 34 does not enforce uniqueness so duplicate-first behaviour is explicitly documented. A future attacker who somehow inserts a colliding email needs write access to auth.json which is a higher boundary than this lookup. |
