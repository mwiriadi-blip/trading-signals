<objective>
Wire `user_id` into the cookie/middleware layer per D-03/D-04/D-05/D-06, with the four `--reviews` consensus fixes applied:

1. **Trusted-device decision (HIGH, Option B):** The `tsi_trusted` path stays at `user_id = None` (admin using a trusted device CANNOT access /admin/* until they re-authenticate via full TOTP flow). This regression is documented as intentional via (a) a `logger.warning` emitted on every trusted-device session, (b) a dedicated test `test_trusted_device_admin_access_returns_403` that asserts this behaviour, and (c) explicit acknowledgement in the threat model. **Option A (resolving uid in Path 2) was rejected** because trusted-device cookie payload is `{uuid, iat}` with NO username field — no way to map device→admin without introducing a fragile env-var lookup that would mask future multi-user trusted-device scope.

2. **Default ordering (MEDIUM, Codex+OpenCode):** `request.state.user_id = None` is set at the VERY TOP of `AuthMiddleware.dispatch` — first statement after the docstring/method signature, BEFORE the PUBLIC_PATHS check, BEFORE `_try_cookie`, BEFORE any conditional. Documented inline; tested explicitly.

3. **Shim observability (LOW, Gemini+OpenCode):** When D-04 shim resolves uid from username via `get_user_by_email`, emit `logger.info('[Auth] D-04 cookie shim resolved uid for uname=<X>')`. Also test the shim path explicitly (cookie with `u` but no `uid` → request.state.user_id resolves to admin uid).

4. **Cookie payload + lookup:** _make_session_cookie(uname, uid=None) serialises {'u': uname, 'uid': uid, 'iat': ...} and both callers (post_enroll, post_verify) look up admin uid via auth_store.get_user_by_email(username) before issuance.

Purpose: Plan 03's `current_user_id` and `require_admin` factories rely on `request.state.user_id` being populated by middleware. Without this plan the dependency layer cannot read user identity.

Output: One cookie-payload extension + one middleware extension with reordered default + two warning/info log lines + two new test classes.

**Existing routes migration scope (cross-cutting --reviews HIGH decision):** The factory `Depends(current_user_id)` is created in Plan 03 but NOT injected into existing non-admin authenticated route handlers (`/trades`, `/journal`, `/dashboard`, etc.) in this phase. Phase 35's goal is "admin remains the only user — no observable behaviour change." Migrating existing routes is **deferred to Phase 36** (per-user scoping). Phase 35 satisfies RBAC-01 partially — the factory exists, the cookie carries `uid`, middleware populates `request.state.user_id`, and admin-gated routes (only /admin/ping today) consume the factory. Phase 36 will wire `Depends(current_user_id)` into the existing authenticated route handlers. Plan 05 acceptance criteria reflect this scope.
