<objective>
Wire `user_id` into the cookie/middleware layer per D-03/D-04/D-05/D-06. Two production changes: (1) `_make_session_cookie(uname, uid=None)` serialises `{'u': uname, 'uid': uid, 'iat': ...}` and its two callers (`post_enroll`, `post_verify`) look up the admin's uid via `auth_store.get_user_by_email(username)` before issuance; (2) `AuthMiddleware.dispatch` sets `request.state.user_id = None` as a default at the top of the auth section, and `_try_cookie` Path 1 (tsi_session) sets `request.state.user_id` from `payload['uid']` directly on the happy path or via the D-04 shim (`get_user_by_email(payload['u'])`) when uid is absent/None.

Purpose: Plan 03's `current_user_id` and `require_admin` factories rely on `request.state.user_id` being populated by middleware. Without this plan the dependency layer cannot read user identity. The shim ensures pre-Phase-35 cookies (no `uid` field) and the bootstrap-gap case (admin not in `users[]` yet) still decode without forcing a re-login.

Output: One cookie-payload extension + one middleware extension + two new test classes. No observable behaviour change for admin (still admin-only, no role checks yet — those land in Plan 03).
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-02-01 | Spoofing | tsi_session payload | mitigate | itsdangerous URLSafeTimedSerializer signature already enforced on every decode (existing); uid is part of the signed payload — tampered uid invalidates signature and falls through to BadSignature. |
| T-35-02-02 | Elevation of privilege | uid manipulation in cookie | mitigate | Even with a signed cookie, the D-07 require_admin (Plan 03) reads role from auth.json live via get_user(uid) — cookie cannot smuggle role=admin; only a real users[] row with role='admin' grants access. |
| T-35-02-03 | Information disclosure | shim get_user_by_email lookup | accept | Lookup happens server-side only; the resolved uid is set on request.state and never echoed in response bodies. No PII leak. |
| T-35-02-04 | Tampering | Pitfall 3 — user_id leak across paths | mitigate | `request.state.user_id = None` set at top of dispatch BEFORE any auth path runs; absence is unambiguous; no cross-path leak. |
| T-35-02-05 | Spoofing | _try_header sets user_id=None | mitigate (intentional) | Header auth is operator/script path; intentionally cannot reach /admin/* — D-06 by design. |
| T-35-02-06 | Denial of service | shim auth.json read per stale cookie | accept | Shim path runs at most once per request; auth.json is small (single admin user); load_auth is already fast. The shim is a grace-period mitigation — once admin is bootstrapped + logs in once, the happy path (no auth.json read) dominates. |
