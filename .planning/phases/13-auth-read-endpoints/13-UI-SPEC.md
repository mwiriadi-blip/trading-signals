# Phase 13 — UI Design Contract

> Visual and interaction contract. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase Classification

**Passthrough phase — no new visual design.**

Phase 13 is a server-side auth + read-endpoint layer. It does not invent, ship, or modify any visual component. The sole "UI" artefacts are four HTTP response contracts (byte-level):

1. `GET /` — serves the existing v1.0 `dashboard.html` via `FileResponse`
2. `GET /api/state` — returns JSON (no UI)
3. 401 response body (plain text)
4. 503 "first-run" response body (plain text)

All visual design (color, typography, spacing, layout, Chart.js equity curve, palette tokens in `system_params._COLOR_*`, accessibility posture, responsive posture, dark mode) is inherited from v1.0 `dashboard.py`. Phase 13 is a byte-for-byte passthrough of that artefact.

The orthodox UI-SPEC sections (Design System / Spacing / Typography / Color / Copywriting / Registry) are therefore intentionally minimal below. Inventing tokens for a phase that doesn't render anything would be dishonest padding.

**Anything visual in the dashboard itself belongs to v1.0 `dashboard.py` + `render_dashboard()`. Phase 13 is not authorised to change it.** Future phases that DO change the dashboard (Phase 14 HTMX forms, Phase 15 calculator + sentinel banners) will write their own UI-SPEC against v1.0 as baseline.

| Property | Value | Source |
|----------|-------|--------|

## Copywriting Contract

Two operator-facing strings ship in this phase. Both are intentionally terse, machine-readable, lowercase, and contain zero actionable guidance (by design — AUTH-02 "no hints" and D-10 "minimal"). A human browsing the site without the secret sees `unauthorized`; a human browsing after fresh install before cron runs sees `dashboard not ready`. Both are expected to be debugged via journald, not inferred from the body.

| Element | Copy | Rationale |
|---------|------|-----------|
| 401 response body | `unauthorized` | Verbatim match to REQUIREMENTS.md AUTH-02 and D-04. Must not be changed — checker asserts literal equality. |
| 503 first-run body | `dashboard not ready` | Verbatim match to D-10. Describes the condition in three plain-English words without leaking internal mtime semantics. Checker asserts literal equality. |
| Primary CTA | n/a | Phase 13 has no CTA — read-only endpoints. |
| Empty-state heading | n/a | Dashboard empty state (no positions, no trades) is v1.0's concern. |
| Empty-state body | n/a | Same as above. |
| Error state (dashboard render failed mid-hit) | (no body copy — user sees the stale dashboard; the failure surfaces only in journald via `[Web] dashboard regen failed, serving stale: ...`) | D-10. Never-crash posture: user never sees an error page. |
| Destructive confirmation | n/a | No mutations in this phase. |

**No translation, no i18n, no styling.** These bodies are plain ASCII. No punctuation, no trailing newline, no framing (`Error:` / `Status:` prefixes are forbidden).

## Accessibility Passthrough Note

**Phase 13 does not audit, change, or improve the dashboard's accessibility.** v1.0 `dashboard.py` is the sole authority for:

- Semantic HTML structure (landmarks, headings, tables)
- Color contrast (palette tokens in `system_params`)
- Focus management, keyboard navigation
- Screen-reader labelling of Chart.js equity curve

If the v1.0 dashboard has accessibility gaps, they ship unchanged through this phase. Any a11y improvement belongs in a dedicated future phase with explicit scope (not v1.1 per REQUIREMENTS.md §Future "Mobile-responsive dashboard layout refinements"-adjacent work).

The 401 and 503 plain-text responses are trivially accessible by nature (no markup, no interactive elements).

## Browser / Client Compatibility Passthrough Note

v1.0 `dashboard.html` already targets its own browser matrix (Chart.js 4.4.6 UMD via jsDelivr with SRI). Phase 13 inherits that matrix without modification. `curl`, `httpie`, and any JSON consumer consuming `/api/state` sees a standard `application/json` body with no browser-specific quirks.

The `FileResponse` ETag/Last-Modified machinery enables conditional GETs (`If-None-Match`, `If-Modified-Since`) for 304 responses — this is a free win from Starlette, not a Phase 13 design decision. Planner and executor do NOT need to hand-roll this; just use `FileResponse` per D-07.
