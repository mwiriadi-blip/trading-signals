# Phase 15 — UI Design Contract

> Visual and interaction contract. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Classification

**Enhancement phase — extends the v1.0/Phase 14 dashboard with two new surface families.**

Phase 14 added mutation forms and action buttons. Phase 15 adds:

1. **Calculator rows (CALC-01..04):** Per-position rows gain a second "calculator sub-row" showing trailing stop details, distance-to-stop, next pyramid trigger, forward-looking peak-stop input, and an entry-target block when signal=LONG/SHORT but position=FLAT.
2. **Sentinel banners (SENTINEL-01..03):** Drift/reversal banners on the dashboard above the Open Positions section; the same banners surface in the daily email as top-tier critical banners.

All other dashboard and email surfaces are untouched. Phase 15 has no licence to change existing v1.0 formatting, the equity chart, closed-trades table, key stats, or footer.

**Inheritance discipline.** The design system in `dashboard.py::_INLINE_CSS` and `system_params._COLOR_*` remains the sole source of truth. Phase 15 adds new CSS class rules that compose the existing tokens — never new color or type tokens. Phase 14 introduced `.banner-success` and `.badge-manual` as the CSS extension model; Phase 15 follows the same extend-only pattern.

**v1.0 / Phase 14 aesthetic constraints carried into Phase 15:**
- Dark dashboard (`_COLOR_BG = #0f1117`)
- Terminal-like density (compact rows, monospace numbers, no chrome bloat)
- HTMX for the forward-look fragment GET; no additional JS beyond what is already loaded
- Auth gate: `X-Trading-Signals-Auth` header on every HTMX request via existing `hx-headers`
- No React, no build step, no icon library

|------|------|--------|-------------|----------------|

**60/30/10 audit:**
- 60% (`_COLOR_BG #0f1117`) — page surface and banner surround
- 30% (`_COLOR_SURFACE #161a24` + `_COLOR_BORDER #252a36`) — calculator sub-row background, banner fill, entry-target block
- 10% (the three accents) — strictly bound to the explicit allowlist below

**Accent reserved for (explicit allowlist — never broaden):**
- `_COLOR_LONG` (#22c55e): Entry-target block direction label when signal is LONG; LONG direction inline in drift banner per-instrument lines
- `_COLOR_SHORT` (#ef4444): Reversal banner 4px left border; SHORT direction inline in drift banner per-instrument lines
- `_COLOR_FLAT` (#eab308): Drift banner (severity='drift', no reversal) 4px left border; `manual:` label prefix in the side-by-side computed/manual stop cell

**Banner border logic:**
- Any `DriftEvent` with `severity='reversal'` present → banner border `_COLOR_SHORT` (red). Even if other events are severity='drift'.
- All events are severity='drift' (none are reversals) → banner border `_COLOR_FLAT` (amber).
- This matches D-13 severity hierarchy: reversal overrides drift color for the combined banner.

**Email banners use inline-CSS versions of the same tokens** (no CSS variables in email HTML — must be hex literals, same as the Phase 8 stale/corruption banners in `notifier.py`).

## Interaction Contract

### Forward-look HTMX (CALC-03)

**Trigger:** `hx-trigger="input changed delay:300ms"` on the `<input type="number">` in `.calc-forward`.

**Endpoint:** `GET /?fragment=forward-stop&instrument={SPI200|AUDUSD}&z={value}` — per Phase 13 D-12 / Phase 14 fragment-GET pattern.

**Target:** `#forward-stop-{instrument}-w` (explicit ID more debuggable than `closest .w-cell` — per 15-CONTEXT.md §Claude's Discretion).

**Response:** Server returns a plain HTML fragment — just the `<span id="forward-stop-{instrument}-w">` element with the W value (or `—` if Z ≤ 0 or not parseable). No wrapping `<div>`, no full `<html>`.

**Auth:** The fragment GET is behind `AuthMiddleware` and MUST carry the `X-Trading-Signals-Auth` header — injected via the existing `hx-headers` on the parent `<tbody>`.

**No-input state:** When the input is empty (Z = ''), the W span shows `—` (em-dash) with the hint text `(enter high to project)` in `--color-text-dim`. No HTMX request fires for empty input.

**Validation:** Server returns `—` (not an error) when Z is ≤ 0 or below position entry price (degenerate input). No 4xx. This prevents the error-handling machinery from triggering on interim partial-typed values.

### Drift banner render trigger

The drift sentinel banner is a **server-side render** (not a client-side dynamic update). It is part of the full dashboard render and also part of the per-instrument `<tbody>` fragment (so a close/modify swap on that tbody refreshes the banner too). No separate client-side polling.

### No new destructive actions in Phase 15

Phase 15 introduces no new writes from the browser (the forward-look GET is read-only). All drift warning writes happen server-side (main.py signal loop and web/routes/trades.py mutators). No new confirmation flows, no new `confirm()` dialogs.

## Banner Stack Hierarchy (D-13)

When multiple critical states coexist, render in this order (most severe first), each as its own distinct banner:

| Priority | Source | Color | Label |
|----------|--------|-------|-------|
| 1 (most severe) | `state['_stale_info']` (transient) | `_COLOR_SHORT` red border | `━━━ Stale state ━━━` |
| 2 | `warnings` where `source='state_manager'` + message starts with `recovered from corruption` | `_COLOR_FLAT` amber border | `━━━ State was reset ━━━` |
| 3 | Any `DriftEvent` with `severity='reversal'` | `_COLOR_SHORT` red border | `━━━ Drift detected ━━━` |
| 4 (least severe) | All `DriftEvent` with `severity='drift'`, no reversal | `_COLOR_FLAT` amber border | `━━━ Drift detected ━━━` |

**Combined drift banner:** When both drift and reversal events exist, they are merged into one banner using the reversal (red) color. There is never more than one drift banner shown at once. The single merged banner lists all affected instruments (one bullet per instrument).

## Accessibility Posture

Phase 15 adds a form input (forward-look) and new table rows.

**Required accessibility behaviours for new Phase 15 surfaces:**

- The forward-look `<input>` MUST have an `aria-label` attribute (there is no visible `<label for="...">` since the input is inline within a table cell). Use `aria-label="Enter today's high to project trailing stop for {instrument}"`.
- The drift banner div MUST have `role="alert" aria-live="polite"` so screen readers announce new banners without page refresh.
- `.calc-row` rows are informational — they do not require `scope` or `<th>` cells, but the entire row MUST have `aria-label="Calculator data for {instrument}"` for screen reader context.
- The `(will close)` annotation in the side-by-side stop cell MUST be wrapped in `<em>` (semantic italic) not CSS `font-style: italic` on a `<span>` — the `<em>` carries semantic emphasis meaning.
- The entry-target block MUST NOT use color alone to convey direction (LONG/SHORT) — the text label is always present alongside the color.

**Out of scope for Phase 15 a11y:**
- WCAG color-contrast audit of the new `--color-text-dim` (#64748b) on `--color-bg` (#0f1117) — this combination is used for de-emphasised informational text already established in v1.0. No change in Phase 15.
- Screen-reader-optimised calendar input for the existing open form (Phase 14 scope).
