<objective>
Two web-layer changes that together close the dashboard interactivity loop:

1. Forward-look HTMX fragment endpoint (CALC-03): Add `?fragment=forward-stop&instrument=X&z=Y` branch to `web/routes/dashboard.py`. Operator types Z (today's high) into the calc-row input shipped in Plan 05; HTMX fires this GET; handler synthesizes a position with updated peak/trough, calls `sizing_engine.get_trailing_stop`, returns a span with the projected stop W. Degenerate Z (empty, ≤0, non-finite) returns em-dash without an error.

2. Drift recompute in mutation handlers (D-02): Each `_apply` mutator in `web/routes/trades.py` (open/close/modify) gains a 3-step block AFTER the position mutation: clear stale drift → detect_drift on the post-mutation state → append fresh drift warnings. After every trade mutation, the next dashboard render shows the correct drift banner (or absence thereof).

Both changes import sizing_engine and state_manager LOCALLY inside the relevant function/closure bodies (C-2 pattern; FORBIDDEN_FOR_WEB AST guard already permits these — Phase 14 D-02 promotion).

Plan 05 already shipped the input markup that calls `?fragment=forward-stop`. Plan 06 ships the endpoint that responds.

Purpose: Make the calc-row interactive (typed Z → projected W in under 500ms) and keep drift state in sync after every mutation. Without this plan, the input does nothing and drift banners go stale immediately after a trade.
Output: ~50 lines new in web/routes/dashboard.py, ~10 lines per mutator (×3) in web/routes/trades.py, plus 12 populated test methods.
