<objective>
Wire the drift detection lifecycle into the daily signal loop. Three things happen in main.py:

1. After the existing `pending_warnings` flush in `run_daily_check`, insert a 3-step block: clear stale drift warnings → call detect_drift → append fresh drift warnings.
2. Emit a `[Sched]` log line per drift event so droplet journald captures the trigger.
3. Preserve the W3 invariant: the mutate_state call count must remain exactly 2 per run (the existing terminal mutate_state call captures all drift mutations because they happen on the same `state` dict that the closure replays).

This is the single hardest plan in Phase 15 because the W3 invariant test is unforgiving. Pitfall 5 in 15-RESEARCH.md is the canonical failure mode: adding a separate mutate_state for drift breaks the W3 lock.

The orchestration sequence in run_daily_check at the end of the run looks roughly:
```
... compute signals/positions ...
... pending_warnings flush ...      <-- Phase 8
[NEW Phase 15 block goes HERE]      <-- in-memory mutations only
state['last_run'] = run_date_iso
_accumulated = state
def _apply_daily_run(fresh_state):
  for key in (...): fresh_state[key] = _accumulated[key]
state = state_manager.mutate_state(_apply_daily_run)   <-- single save (count #1)
```

The drift block must appear AFTER the pending_warnings flush (so any sizing_engine warnings raised during the run are not nuked) and BEFORE `_accumulated = state` (so the captured snapshot includes drift warnings).

Purpose: Make drift warnings part of the normal daily-run state contract. By the time the email dispatch path runs, state['warnings'] already contains the right drift entries.
Output: ~6 inserted lines in run_daily_check + 4 populated test methods that prove the lifecycle and W3 invariant.
