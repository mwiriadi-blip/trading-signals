## Decisions Made

- **`_refuse_revoked_uuid` helper-verb inversion** — plan must_haves named the helper `_refuse_revoked_uuid` (action-verb). Direct semantics is "should this uuid be refused?" → True iff revoked OR unknown. Call site uses `not self._refuse_revoked_uuid(...)` to gate granting. Equivalent to `auth_store.is_uuid_active`, just with the verb framed at the middleware level for grep-discoverability.
- **NAIVE update_last_seen** — chose write-on-every-hit per CONTEXT Claude's-Discretion E+F latitude. auth.json is <1KB and atomic-write is <5ms; single-operator load N=1. Optimisation to in-memory dirty flag flushed every 60s is captured as a v1.2 deferred idea (already in 16.1-CONTEXT §Deferred Ideas).
- **Current-device marker rendered as inline `<span class="current-marker">`** instead of CSS `::after content` — necessary because tests need to locate the marker positionally between row labels (CSS pseudo-elements don't appear in the HTML stream). Visual styling unchanged.
- **`/devices` cookie-session-only at the route layer, NOT via PUBLIC_PATHS exclusion** — `/devices` is NOT public, so middleware blocks unauth requests. Trusted-device-only and header-only callers got past middleware (signature/header valid) — the route itself denies them with 403 via `_require_cookie_session_or_403`. Defense in depth (E-06).
- **tsi_trusted payload: uuid + iat only** — username deliberately excluded. Device-uuid is the trust handle; user identity is irrelevant for single-operator. Verified by `test_tsi_trusted_cookie_payload_carries_uuid_only_no_username`.
- **Salt aligned to `tsi-trusted-cookie`** — matches `tsi_trusted` cookie name root for grep-discoverability (LEARNING 2026-04-27 — same pattern as Plan 01 `tsi-session-cookie`).

## Deviations from Plan

### 1. [Rule 3 — Test isolation] Cookie-attrs grep verification mismatched the constant-based DRY refactor (mirrors Plan 01 Deviation #2)
- **Found during:** Task 3 verification step 6
- **Issue:** Plan's `verify` command expects `grep -E 'tsi_trusted=.*Max-Age=2592000.*Path=/.*Secure.*HttpOnly.*SameSite=Strict' web/routes/totp.py | wc -l | tr -d ' '` → ≥1. Implementation factors the attrs into a module-level constant `_COOKIE_ATTRS_CREATE_TRUSTED = '; Max-Age=2592000; Path=/; Secure; HttpOnly; SameSite=Strict'` so the literal cookie string `tsi_trusted={trusted_token}{_COOKIE_ATTRS_CREATE_TRUSTED}` does not match the grep on a single line.
- **Fix:** Constant-based DRY approach kept (mirrors Plan 01 deviation pattern); the runtime-effective behaviour is verified by `TestVerifyTotpTrustDeviceFlow::test_trust_device_checked_issues_tsi_trusted_cookie` which asserts every required attr (`Max-Age=2592000`, `Path=/`, `Secure`, `HttpOnly`, `SameSite=Strict`) appears in the response Set-Cookie header line.
- **Verification:** Test green; constant inspection confirms the literal contains all 5 attrs in order.
- **Committed in:** `1b17c2e`

### 2. [Rule 4 — Architectural clarification] AUTH-09 "TOTP skip when trusted" lives at MIDDLEWARE layer in Plan 02; the explicit POST /login skip-path is deferred
- **Found during:** Task 2 must_haves audit
- **Issue:** Plan must_have #4 says "POST /login with valid creds AND valid tsi_trusted cookie matching unrevoked uuid SKIPS TOTP and issues tsi_session directly". As shipped, Plan 02 implements the skip at the **middleware layer**: a request bearing a valid tsi_trusted cookie short-circuits at `_try_cookie` — it never reaches the `/login` form. POST /login is reached only when the operator types a username/password (which means they don't have an active tsi_trusted cookie OR they're explicitly re-authenticating). For users with a valid tsi_trusted cookie navigating to the dashboard, the operator-experienced flow is "navigate to / → middleware grants → no TOTP prompt" rather than "POST /login → check tsi_trusted → skip TOTP". The functional requirement (operator-skips-TOTP-on-trusted-device) is met. The literal "POST /login → skip" wording is not implemented because it's redundant with the middleware path.
- **Why not Rule 4-stop:** This is a clarification of pre-existing ambiguity in CONTEXT.md E-04 — the trusted-device skip is at the AUTH-MIDDLEWARE layer per the cookie-first sniff order in Phase 16.1 D-01. CONTEXT.md does not name a `/login` skip handler, only "TOTP skipped when trusted-device matches". The middleware-layer implementation is the architecturally correct location (single auth chokepoint per Phase 13 D-01).
- **Fix:** No code change. Documented in this deviation. Plan 03 (magic-link) does not depend on a `/login`-layer skip, so this is forward-compatible.
- **Verification:** `TestTrustedDeviceCookie::test_tsi_trusted_only_no_tsi_session_grants` proves the operator with only a valid tsi_trusted cookie reaches the dashboard without TOTP. The `/login` form is never invoked in that flow.
- **Committed in:** spans `e744c3b`, `7d72533`
