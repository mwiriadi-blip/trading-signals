## B. 401 contract — WWW-Authenticate header

**Background:** Phase 13 D-04 explicitly forbade `WWW-Authenticate` on 401 responses ("no leaked info; no hints; plain-text body `unauthorized`"). Adding Basic Auth normally requires sending `WWW-Authenticate: Basic realm="..."` to trigger the iOS browser dialog. This area resolved the conflict.

| Option | Description | Selected |
|--------|-------------|----------|
| Browser-conditional WWW-Authenticate | Sniff Sec-Fetch-Mode/Sec-Fetch-Dest/Accept; send WWW-Authenticate only for browser navigations. Curl/scripts get D-04 verbatim. | ✓ |
| Always send WWW-Authenticate | Simpler middleware; leaks "this site uses Basic Auth" to scanners; weakens AUTH-02 spirit. | |
| Never send WWW-Authenticate | Operator types `https://user:secret@signals…` in URL bar manually; preserves D-04 verbatim but worse UX. | |

**User's choice:** Browser-conditional.
**Notes:** This decision was SUPERSEDED by Area D follow-up (UX intersection question) which chose "redirect browsers to /login" — meaning `WWW-Authenticate` is in fact NEVER sent. The dialog system path was eliminated entirely in favor of the form. Reconciliation captured in CONTEXT.md D-04..D-05.

## C. Basic Auth credentials shape + realm

**Background:** Basic Auth requires `username:password`. The existing Phase 13 single secret (`WEB_AUTH_SECRET`) doesn't cleanly map. Need to decide credential structure and realm string (what the iPhone dialog shows).

### C.1 Credentials shape

| Option | Description | Selected |
|--------|-------------|----------|
| Fixed username 'operator' + WEB_AUTH_SECRET as password | Single env var; predictable; clean setup-docs. | |
| Ignore username, only check password | Simplest middleware; awkward setup-docs ("any username works"). | |
| Two env vars: WEB_AUTH_USERNAME + WEB_AUTH_SECRET | Operator picks memorable username via .env. | ✓ |

**User's choice:** Two env vars.
**Notes:** Operator preferred configurability over single-env-var simplicity. Username has no entropy role (password carries Phase 13 D-17's ≥32 char ≈128 bit entropy); username is for memorability. Followed up with username-validation question (C.3 below).

### C.2 Realm string (for WWW-Authenticate header)

| Option | Description | Selected |
|--------|-------------|----------|
| 'Trading Signals' | Matches FastAPI app title; neutral. | ✓ |
| 'signals.mwiriadi.me' | Hostname; explicit but slight domain leakage. | |
| 'Operator login' | Generic / role-focused. | |

**User's choice:** 'Trading Signals'.
**Notes:** Captured for completeness but moot — `WWW-Authenticate` is never sent (per Area B/D reconciliation). Realm value lives in CONTEXT.md D-09 in case future undo-decision starts sending the header.

### C.3 WEB_AUTH_USERNAME validation

| Option | Description | Selected |
|--------|-------------|----------|
| Non-empty only, no length minimum | Username has no entropy role; just non-empty + forbid `:`. | ✓ |
| Same ≥32 char check as the secret | Treat username as second factor of entropy. | |
| ≥3 chars (loose memorability minimum), forbid `:` | Catches typos / placeholder values. | |

**User's choice:** Non-empty only, no length minimum.
**Notes:** Validates at `create_app()` boot per Phase 13 D-16 fail-closed pattern. Forbid `:` (Basic Auth field separator). Documented in CONTEXT.md D-08.

## D. Phase 16.1 scope boundary

**Background:** The todo's lean said "Option 1 (Basic Auth) as v1.1.1 patch, Option 2 (cookie) as v1.2". This area decided whether 16.1 is just Basic Auth or both.

| Option | Description | Selected |
|--------|-------------|----------|
| 16.1 = Basic Auth ONLY; cookie deferred to v1.2 phase | ~1 day of work. 1 PLAN.md. Matches todo's v1.1.1 patch framing. | |
| 16.1 = Basic Auth + Login form + cookie BOTH ship | 2 PLAN.md files. ~3-5 days. End-to-end UX shipped. | ✓ |
| 16.1 = Basic Auth + STUB cookie route (501) | Anti-pattern — stubbed routes rot. | |

**User's choice:** Basic Auth + cookie BOTH ship in this phase.
**Notes:** Operator chose the bigger scope. This triggered cookie sub-questions (D.1..D.4 below).

### D.1 Cookie signing library

| Option | Description | Selected |
|--------|-------------|----------|
| itsdangerous | Mature, single-purpose, ~5 lines to sign/verify. | ✓ |
| Hand-rolled HMAC-SHA256 | Zero new deps; ~20 lines; security-critical hand-rolled crypto. | |
| Starlette SessionMiddleware | Built-in but heavier than needed (request.session dict API). | |

**User's choice:** itsdangerous.
**Notes:** New pinned dep in requirements.txt per CLAUDE.md exact-pin convention. Signing key reuses WEB_AUTH_SECRET (no new key env var needed). Captured in CONTEXT.md D-10.

### D.2 Session TTL

| Option | Description | Selected |
|--------|-------------|----------|
| 12 hours (single trading day) | Login at start of day, expire next morning. Max-Age=43200. | ✓ |
| Session-only (dies when browser tab closes) | Most secure; surprise re-login if Safari swiped away. | |
| 7 days | Lower friction; larger stolen-cookie window. | |

**User's choice:** 12 hours.
**Notes:** Cookie carries iat (issued-at) timestamp inside signed payload as belt-and-suspenders against client-side cookie-attribute tampering. Captured in CONTEXT.md D-11.

### D.3 Auth UX intersection (browser → /login OR Basic dialog)

| Option | Description | Selected |
|--------|-------------|----------|
| Redirect to GET /login (form) — no Basic dialog | 302 → /login. Curl still gets plain 401. Basic Auth still works if explicitly sent. | ✓ |
| Send WWW-Authenticate: Basic on 401 — dialog wins, /login is fallback | iPhone dialog first; /login is a backup link in 401 body. | |
| Both fire concurrently (dialog AND form-link in 401 body) | Maximum flexibility; mildly ugly. | |

**User's choice:** Redirect to /login form.
**Notes:** SUPERSEDED Area B's "browser-conditional WWW-Authenticate" decision. Final state: `WWW-Authenticate` is NEVER sent server-side. Browsers get 302 → /login form (polished UX). Curl/scripts get plain 401. Basic Auth `Authorization:` header still ACCEPTED if explicitly sent (URL-bar `https://user:secret@…` triggers this). Documented in CONTEXT.md D-04..D-07.

### D.4 Logout behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Clear cookie + 'Logged out' confirmation page | Honest about Basic Auth Keychain limitation in note. | ✓ |
| Clear cookie + force-re-auth via 401 (evict Basic creds) | Inconsistent browser behavior; Safari/Chrome differ. | |
| No /logout route at all — cookie just expires after 12h | Defensible but feels incomplete given /login exists. | |

**User's choice:** Clear cookie + confirmation page.
**Notes:** POST /logout (form-button on dashboard). Cookie deletion attrs MUST match creation attrs per global LEARNINGS "Cookie deletion must match cookie creation options". Confirmation page footnote: "Basic Auth credentials may still be cached by your browser — close all Safari tabs to fully clear them." Captured in CONTEXT.md D-12.

### D.5 HTMX header preservation in dashboard.py

| Option | Description | Selected |
|--------|-------------|----------|
| Keep header AND rely on cookie — belt-and-suspenders | Zero diff to dashboard.py / web/routes/dashboard.py placeholder substitution. | ✓ |
| Remove header from HTMX forms — cookie-only for browsers | Cleaner; breaks AUTH-05 contract; deletes tested Phase 14 mitigation. | |
| Keep header + add CSRF-token meta tag | SameSite=Strict already blocks CSRF; over-engineered. | |

**User's choice:** Keep header AND cookie.
**Notes:** Zero modifications to Phase 14 Plan 14-04 Task 5 (`web/routes/dashboard.py:73..76, 195..200`) or its threat T-14-15 mitigation tests. AUTH-05 preserved verbatim. Captured in CONTEXT.md D-13.
