**Audit-log pattern** (lines 69..84 — verbatim, Phase 16.1 D-05 inherits):
```python
@staticmethod
def _log_failure(request: Request) -> None:
  xff = request.headers.get('x-forwarded-for', '')
  client_ip = (
    xff.split(',')[0].strip()
    if xff
    else (request.client.host if request.client else '-')
  )
  ua = (request.headers.get('user-agent') or '')[:UA_TRUNCATE]
  logger.warning(
    '[Web] auth failure: ip=%s ua=%r path=%s',
    client_ip, ua, request.url.path,
  )
```

**16.1 reuses verbatim:**
- `EXEMPT_PATHS = frozenset({'/healthz'})` (Phase 13 D-02 — do NOT add `/login` to this set; per CONTEXT integration-points the route is publicly accessible via different mechanism — see `web/routes/login.py` analog below).
- The `_log_failure` method shape (UA truncation, XFF first-entry, %r escaping). Per Open Question #4 in RESEARCH, optionally extend the log line with `reason=basic_bad/cookie_expired/header_missing` as a kwarg with default value `'header_missing'` to preserve Phase 13 grep-test backward compat.
- `hmac.compare_digest(...)` for both username AND secret comparison (CONTEXT D-08 + AUTH-06).

**16.1 changes:**
- Constructor signature: `__init__(self, app, *, secret: str, username: str)` — adds `self._username_bytes = username.encode('utf-8')` alongside the existing `self._secret_bytes`. Threaded by `web/app.py::create_app` per `web/app.py:114` line.
- New helpers `_try_basic`, `_try_cookie`, `_try_header`, `_is_browser_navigation` (RESEARCH provides full code in §HTTP Basic Auth and §Sec-Fetch-* sections — copy verbatim).
- `URLSafeTimedSerializer(secret_key=secret, salt='tsi-session-cookie')` instantiated ONCE in `__init__` (Phase 13 D-18 "secret read ONCE at module load" generalizes here — pre-build the serializer at construction time, never inside `dispatch`).

**Pattern landmines (executor MUST NOT trip):**
1. **Helper-shadowing regression** (RESEARCH §LEARNINGS 2026-04-25). After extracting `_try_header`, the OLD inline header-check code MUST be deleted, NOT preserved alongside. Verification: `grep -n 'compare_digest' web/middleware/auth.py` returns ONE call site (inside `_try_header`), not two.
2. **`_log_failure` exactly-once** (RESEARCH §Sampling Pyramid 1). Helpers MUST return `bool` only — never call `_log_failure` themselves. Only the dispatch driver logs in step 4. Verification: parametrized test at `tests/test_web_auth_basic.py` asserts `len(failure_lines) == 1` per failed request regardless of which combo of headers/cookies are wrong.
3. **`SignatureExpired` MUST come BEFORE `BadSignature` in except-chain** — `SignatureExpired ⊂ BadSignature` per RESEARCH exception hierarchy. LEGB rule: narrower clause first, else expired cookies report as `bad_sig`.
4. **No `WWW-Authenticate` header EVER** (CONTEXT D-04, D-05). Existing Phase 13 test `tests/test_web_auth_middleware.py::TestUnauthorizedResponse::test_no_www_authenticate_header` MUST keep passing. Do NOT add the header in any new branch.
5. **Cookie validation salt** must be `'tsi-session-cookie'` (RESEARCH Open Question #5 reconciles CONTEXT D-10's `'web-auth-cookie'` typo with cookie name `tsi_session` for grep-ability).

def _read_auth_secret() -> str:
  '''D-16/D-17: fail-closed if WEB_AUTH_SECRET is missing or too short.

  Called once at create_app() time. systemd's Restart=on-failure surfaces
  the RuntimeError in journald so the operator sees the cause immediately.

  secret = os.environ.get('WEB_AUTH_SECRET', '').strip()
  if not secret:
    raise RuntimeError(
      'WEB_AUTH_SECRET env var is missing or empty — refusing to start. '
      'Add WEB_AUTH_SECRET=<32+ chars> to /home/trader/trading-signals/.env'

    raise RuntimeError(
      f'WEB_AUTH_SECRET must be at least {_MIN_SECRET_LEN} characters. '
      'Generate with: openssl rand -hex 16'
    )
  return secret

**Pattern landmines:**
1. **GET /login MUST be publicly accessible** — per CONTEXT integration-points the middleware lets `/login` GET through without auth (otherwise login is impossible). Implementation choice (planner): either add a route-aware check inside `_is_browser_navigation` (return special "go to /login" branch when path == '/login') OR add a `EXEMPT_PUBLIC_GET_PATHS = frozenset({'/login'})` allowlist parallel to EXEMPT_PATHS. Either way: POST /login is also publicly accessible (auth bootstrap), but POST /logout REQUIRES auth (only authenticated users can log out).
2. **Cookie attrs deletion must match creation** (RESEARCH §Cookie hygiene + global LEARNINGS direct cite). 4. **Generic "Sign in failed" error** (CONTEXT D-14 + AUTH-02 spirit) — same string + same response timing whether wrong username or wrong password. 5. **`Response` for 302 NOT `RedirectResponse`** — RESEARCH §Starlette middleware "Constructing 302 / 401 responses inline": `RedirectResponse` defaults to 307 (preserves request method — wrong for GET-after-failed-POST). Use `Response(status_code=302, headers={'Location': ...})` explicitly.
