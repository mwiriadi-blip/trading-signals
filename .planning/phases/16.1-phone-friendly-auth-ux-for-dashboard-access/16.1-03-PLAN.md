Rate-limiter — Plan 03 ships in-process token-bucket via auth_store.py helper (F-08 says NOT a new dep):
- Storage: `auth.json` does NOT hold rate state (would couple low-rate config writes with high-rate request bookkeeping). Instead use a process-local `dict[str, list[float]]` keyed by `(ip, path)` → list of timestamps; helper `_check_rate_limit(key, max_count, window_seconds)` lives in `web/middleware/auth.py` (or a sibling helper module). Counts are lost on process restart — acceptable for single-droplet single-operator. Per-account magic-link count (3 per 24h) DOES use auth.json (data already there for audit anyway).
- Implementation lives entirely in middleware module to avoid scattering across routes. Routes invoke `request.state.rate_limit_check(name)` if integration via Starlette state is cleanest, OR middleware short-circuits before routes when over-budget. Pick the simpler in-middleware short-circuit: middleware checks rate limit AFTER PUBLIC_PATHS check but BEFORE the cookie/header sniff for `/login`, `/forgot-2fa`, `/verify-totp`, `/reset-totp` paths.

Magic-link token construction (F-02):
```python
import secrets, hashlib, time
serializer = URLSafeTimedSerializer(WEB_AUTH_SECRET, salt='magic-link')
token = serializer.dumps({'purpose': 'totp-reset', 'iat': int(time.time())})
token_hash = hashlib.sha256(token.encode('utf-8')).hexdigest()
auth_store.add_magic_link(token_hash, action='totp-reset', expires_at=<now+3600>)
# email link:
link = f'https://signals.<owned-domain>.com/reset-totp?token={token}'
# server side validation:
def consume_magic_link(unhashed_token: str) -> tuple[bool, str | None]:
  token_hash = hashlib.sha256(unhashed_token.encode('utf-8')).hexdigest()
  data = load_auth()
  for row in data['pending_magic_links']:
    if row['token_hash'] == token_hash:
      if row['consumed']: return (False, None)
      if datetime.fromisoformat(row['expires_at']) < datetime.now(timezone.utc): return (False, None)
      row['consumed'] = True
      row['consumed_at'] = datetime.now(timezone.utc).isoformat()
      save_auth(data)
      return (True, row['action'])
  return (False, None)
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator's email inbox → magic-link click | Token in URL crosses email-transport (TLS in-transit per Resend) and operator's email client → browser address bar → server |
| Operator's `.env` → OPERATOR_RECOVERY_EMAIL | Boot-time validation; failure → systemd Restart=on-failure surfaces in journald |
| Resend API → notifier.py | Outbound HTTPS POST to api.resend.com; HTTP failures non-crashing |
| auth.json `pending_magic_links` → on-disk hashed tokens | sha256 of token stored, not the token itself; leaked auth.json doesn't expose live tokens |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-16.1-20 | Spoofing | Forged magic-link via signature-replay | mitigate | itsdangerous HMAC signature with WEB_AUTH_SECRET (≥32 char); separate salt `'magic-link'` ensures token cannot be replayed against tsi_session/tsi_pending/tsi_enroll/tsi_trusted serializers (RESEARCH §itsdangerous salt-non-optional) |
| T-16.1-21 | Information Disclosure | /forgot-2fa response shape leaks credential validity | mitigate | E-07 + F-08: response is ALWAYS the same generic 'Check your email' page regardless of credential validity; same response timing within ±10ms (achieved by always doing `hmac.compare_digest` even on credential-mismatch path before returning) |
| T-16.1-22 | DoS / Brute-force | Rapid /forgot-2fa POSTs to enumerate accounts via timing | mitigate | F-08: 3 attempts per hour per IP; constant-time response shape; magic-link generation only on full-cred match (no per-step timing leak) |
