<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public Internet → POST /login | Untrusted form-encoded body (`username`, `password`) crosses here; `?next=` query param hostile |
| Browser cookie storage → AuthMiddleware._try_cookie | Cookie value is operator-controllable (DOM access via document.cookie blocked by HttpOnly, but raw HTTP request can carry any cookie) |
| Operator's session lifetime → cookie revocation | Cookies are stateless; once issued, valid until 12h TTL regardless of /logout — documented in code |
| Form `?next=<path>` → 302 Location | Open-redirect classic vulnerability surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16.1-08 | Tampering | Cookie payload modified by attacker | mitigate | itsdangerous HMAC signature; `BadSignature` exception → fall through to 302/401; SignatureExpired clause MUST come BEFORE BadSignature in except-chain (LEGB rule per RESEARCH §itsdangerous exception hierarchy) |
| T-16.1-09 | Spoofing | Session fixation (attacker pre-sets cookie before login) | mitigate | New cookie issued on EVERY successful POST /login (no token reuse); existing cookie ignored if creds re-validated |
| T-16.1-10 | Tampering | Open-redirect via `?next=//evil.com` | mitigate | `_is_safe_next` validator with 6 distinct guards (RESEARCH §Open-redirect prevention lines 408..458); 14+ bypass payloads in `tests/test_web_routes_login.py::test_open_redirect_blocked` parametrize; failures redirect to `/` |
| T-16.1-11 | Repudiation | Cookie validation silently swallowing errors without logging | mitigate | `SignatureExpired`, `BadSignature`, malformed-cookie all fall through to step 4 of dispatch which logs ONCE per request via `_log_failure` (sampling pyramid 1 invariant from Plan 01 preserved) |
| T-16.1-12 | Information Disclosure | Login form distinguishing wrong-username vs wrong-password | mitigate | Generic "Sign in failed" client-facing message (D-14); server-side log line CAN distinguish (`reason=wrong_username`/`reason=wrong_secret`) for audit utility — does not leak to client. Both failure modes have identical response timing (no early-return short-circuit on username mismatch) |
| T-16.1-13 | Tampering | CRLF response splitting via `?next=/path\r\nSet-Cookie:evil=1` | mitigate | `_is_safe_next` blocks all control chars `ord(c) < 0x20` and `\x7f` (RESEARCH §Open-redirect line 456) |
| T-16.1-14 | Spoofing | CSRF on POST /login | accept | SameSite=Strict cookie + form submits to same origin = no CSRF surface for single-operator (CONTEXT D-13). No CSRF token machinery needed |
| T-16.1-15 | Information Disclosure | Cookie revocation attack — stolen cookie remains valid until 12h TTL even after operator logs out | accept | Documented in code via sampling pyramid 2 test `test_logout_does_NOT_invalidate_existing_cookie_value` — counter-intuitive assertion locks the architectural limitation. v1.2 candidate for server-side revocation list (Redis or on-disk JSON). Single-operator scope makes acceptable risk |
| T-16.1-16 | Tampering | Static asset middleware ordering — login form CSS returns 401 if static mount happens after AuthMiddleware | mitigate | Plan 02 explicitly does NOT mount static files (CSS is inline `_LOGIN_INLINE_CSS` constant per UI-SPEC §"Out of Scope"); `web/static/` deferred to v1.2 |

T-16.1-14 and T-16.1-15 are explicitly accepted with documented rationale. All other threats → BLOCKING with mitigations above.
