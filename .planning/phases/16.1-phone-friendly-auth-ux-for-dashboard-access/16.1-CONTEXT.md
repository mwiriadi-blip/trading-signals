# Phase 16.1: Phone-friendly auth UX for dashboard access - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Add browser-friendly authentication paths to the dashboard so the operator can use `https://signals.<owned-domain>.com/` from a real iPhone (Safari + Chrome, no extension installed) without weakening the existing `X-Trading-Signals-Auth` header gate that curl/scripts and Phase 14 HTMX forms depend on.

After this phase, three valid auth paths exist:

1. **Header path (existing — Phase 13 D-01..D-06):** `X-Trading-Signals-Auth: <secret>` — used by `curl`, scripts, and Phase 14 HTMX trade-mutation forms. Unchanged.
2. **Basic Auth path (new — Plan 16.1-01):** `Authorization: Basic base64(username:secret)` — fallback for power-users who type `https://user:secret@signals…` in the URL bar; iOS Safari accepts this format and stores creds in Keychain. The system Basic-Auth dialog is **not** triggered server-side (no `WWW-Authenticate` header sent).
3. **Cookie path (new — Plan 16.1-02):** `GET /login` renders an HTML form; `POST /login` validates `username + secret`; success sets a 12-hour `itsdangerous`-signed `Secure; HttpOnly; SameSite=Strict` cookie. `POST /logout` clears the cookie and renders a confirmation page. Browser navigations without auth get **302 → /login** instead of `401 unauthorized`.

**Phase 16.1 requirements (4):** AUTH-04 (iOS-native operator auth UX, no extension), AUTH-05 (new path additive — header path unchanged, HTMX forms keep working), AUTH-06 (auth strength preserved — env-only secret, `hmac.compare_digest`, single-operator scope), AUTH-07 (`/healthz` exempt; curl 401 contract preserved verbatim for non-browser requests).

**Explicitly out of scope (deferred to future phases / v1.2):**
- Cloudflare Access / Zero Trust (Option 3 from todo) — would require re-orange-clouding DNS + rebuilding Let's Encrypt as DNS-01 (currently HTTP-01 since DNS is grey-clouded per UAT-16-A 2026-04-27). Architectural change; out of scope for an auth-UX phase.
- Magic link via Resend (Option 4 from todo) — requires the cookie-session machinery from Plan 16.1-02 PLUS email send-on-login. Heaviest UX. Captured in deferred for v1.2 if cookie+passwordless-flow becomes desired.
- Multi-user / OAuth / passkeys / WebAuthn — hard constraint per PROJECT.md "single-operator tool".
- Login-attempt rate limiting at nginx layer for `/login` — defense-in-depth follow-up; same rationale as Phase 13 D-Claude's-Discretion (rate-limit on `/` and `/api/state`). Captured in deferred; lift if `[Web] auth failure:` log noise grows.
- CSRF token machinery — `SameSite=Strict` cookie + single-form-per-site means CSRF surface is essentially nil. Skipped per Area D follow-up.

**Parallelizable with Phase 16?** Yes — Phase 16 (UAT completion) is operationally weekday-gated (UAT-16-B/C wait for a real drift email to land in Gmail). Phase 16.1 only needs Phase 13's `AuthMiddleware` to extend (loosened from "Depends on: Phase 16" to "Depends on: Phase 13" per ROADMAP.md insertion-commit note 2026-04-27).

## Implementation Decisions

### Area A — Mechanism choice (LOCKED)

- **D-01: Phase 16.1 ships TWO new auth paths in one phase: Basic Auth (parallel to header) AND login form + signed cookie session.** Both layer onto Phase 13's `AuthMiddleware` (single ASGI chokepoint per Phase 13 D-01). Plan structure: 2 PLAN.md files. `16.1-01` = Basic Auth wave (~20-line middleware extension + env vars + setup-docs). `16.1-02` = login form + cookie wave (depends on 01; reuses the middleware helper that decodes Basic to also validate the cookie). Cookie middleware extension layers on Basic Auth's middleware diff. Both paths additive — header path (Phase 13 D-01..D-06) unchanged.

- **D-02: Cloudflare Access (todo Option 3) and magic link (todo Option 4) are deferred to v1.2.** Cloudflare Access requires re-orange-clouding (DNS is grey-clouded per UAT-16-A 2026-04-27 — `no Cloudflare proxy in path`) + rebuilding LE renewal as DNS-01 + creating Cloudflare API token + installing certbot-dns-cloudflare. Architectural change, not an auth-UX patch. Magic link is heaviest UX (must check email on first visit) and requires the cookie-session machinery underneath anyway. Captured in `<deferred>`.

- **D-03: AUTH-05 contract preserved verbatim.** All three paths must coexist — adding a new path NEVER removes or weakens the existing header path. `curl -H "X-Trading-Signals-Auth: <secret>" https://signals.<owned-domain>.com/` returns 200 with full dashboard HTML (test asserts byte-equality with pre-16.1 baseline if practical). HTMX forms in `dashboard.py` keep emitting `hx-headers='{"X-Trading-Signals-Auth": "{{WEB_AUTH_SECRET}}"}'` placeholder — D-13 below.

### Area B — 401 contract reconciliation (LOCKED, RECONCILES Phase 13 D-04)

- **D-04: Browser-conditional behavior is `302 → /login` redirect, NOT `WWW-Authenticate: Basic`.** A browser request (sniff: `Sec-Fetch-Mode: navigate` OR `Sec-Fetch-Dest: document` OR `Accept: text/html`) without valid auth returns `302 Location: /login?next=<original_path>`. The Basic Auth system dialog is never triggered server-side. Rationale: forms (D-09..D-14) deliver a polished UX (logo placeholder, friendly error messages, branded styling) that the browser system dialog cannot match; eliminating the dialog also avoids the `https://user:pw@host/` URL-bar ugliness.

- **D-05: Phase 13 D-04 letter relaxed only for browser navigation.** D-04's "no leaked info; no hints; plain-text body `unauthorized`" still holds for ALL non-browser requests:
  - `curl` without `Accept: text/html` and without `Sec-Fetch-Mode` headers → `401 unauthorized` plain-text, no `WWW-Authenticate` header, no redirect.
  - Scripts / API clients / HTMX XHR (XMLHttpRequest carries `Sec-Fetch-Mode: cors` or `same-origin`, NOT `navigate`) → `401 unauthorized` plain-text, no redirect.
  - Browser navigation (Sec-Fetch-Mode: navigate OR Accept: text/html via address-bar paste / link click) → `302 → /login`.

- **D-06: Basic Auth `Authorization: Basic ...` header is ACCEPTED if explicitly sent**, but the system dialog is never triggered. Operator can use `https://user:secret@signals.mwiriadi.me/` URL-bar form (Safari accepts, stores in Keychain). Curl `--user user:secret` works. The middleware decodes the Basic header, splits on first colon, validates `username == WEB_AUTH_USERNAME` AND `hmac.compare_digest(password.encode(), WEB_AUTH_SECRET.encode())`. On match → grant; on mismatch → fall through to whichever 302/401 path Sec-Fetch sniffing chose.

- **D-07: Sniff order in middleware** (after `EXEMPT_PATHS` per Phase 13 D-02):
  1. Decode `Authorization: Basic <b64>` if present → if valid → grant.
  2. Validate signed cookie (`tsi_session=<itsdangerous-token>`) if present → if valid AND not expired → grant.
  3. Validate `X-Trading-Signals-Auth` header if present → if `hmac.compare_digest` match → grant (Phase 13 D-03 unchanged).
  4. None matched → log failure (Phase 13 D-05 audit log unchanged) → branch on Sec-Fetch:
     - Browser navigation → `302 Location: /login?next=<original_path>` (D-04).
     - Anything else → `401 unauthorized` plain-text, no `WWW-Authenticate` (D-05).

### Area C — Credentials shape, env vars, validation (LOCKED)

- **D-08: TWO env vars required: `WEB_AUTH_USERNAME` and `WEB_AUTH_SECRET`.** Both validated at `create_app()` boot per Phase 13 D-16/D-17 fail-closed pattern (RuntimeError → systemd Restart=on-failure surfaces in journald). `WEB_AUTH_USERNAME` validation: non-empty (`len(username.strip()) > 0`), no length minimum, must NOT contain `:` (Basic Auth field separator — `:` in username breaks decode). `WEB_AUTH_SECRET` validation: existing Phase 13 D-17 ≥32 char check stays. Setup-docs (Phase 13 SETUP-DROPLET.md "Configure auth secret" section) gets a sibling step: pick a memorable username (`marc`, `operator`, etc.) — username has no security-entropy role here (password carries 128+ bits via D-17), so any printable non-`:` string is fine.

- **D-09: Realm string `"Trading Signals"` captured for completeness, but moot.** Since `WWW-Authenticate` is never sent (D-04..D-05), realm only appears if a future undo-decision changes course. If we ever start sending `WWW-Authenticate: Basic`, the realm value is `"Trading Signals"` (matches FastAPI app `title='Trading Signals'` in `web/app.py:88`).

### Area D — Login form + cookie session details (LOCKED)

- **D-10: Cookie crypto = `itsdangerous` `URLSafeTimedSerializer`.** New pinned dependency in `requirements.txt` (specific version pinned per CLAUDE.md §Stack convention — planner picks current latest stable, no `>=`). Mature single-purpose library (Pallets/Flask author), well-audited HMAC, built-in expiry checks via `loads(..., max_age=N)`, constant-time signature verification under the hood. ~5 lines to sign/verify in `web/middleware/auth.py`. Signing key is `WEB_AUTH_SECRET` (no new key env var — same secret already validated as ≥32 char per D-08).

- **D-11: Session TTL = 12 hours (`Max-Age=43200`).** Cookie is `Set-Cookie: tsi_session=<token>; Max-Age=43200; Secure; HttpOnly; SameSite=Strict; Path=/`. Operator logs in at start of trading day; session expires by next morning. iPhone Safari Keychain auto-fills cached creds on next-day re-login (one tap). The 12h TTL is also encoded inside the signed payload as `iat` (issued-at) timestamp, so `URLSafeTimedSerializer.loads(..., max_age=43200)` cross-checks the cookie-attribute Max-Age. Belt-and-suspenders against client-side cookie-attribute tampering.

- **D-12: `POST /logout` clears cookie + renders 'Logged out' confirmation page.** Endpoint accepts POST only (form-button on dashboard). Sets `Set-Cookie: tsi_session=; Max-Age=0; Path=/; Secure; HttpOnly; SameSite=Strict` (cookie-deletion attrs MUST match cookie-creation attrs per global LEARNINGS "Cookie deletion must match cookie creation options"). Renders an HTML page: title "Signed out", body "You're signed out. <a href='/login'>Sign in again</a>." with a footnote: "Basic Auth credentials may still be cached by your browser — close all Safari tabs to fully clear them." Honest about the iOS Keychain limitation. NO best-effort 401-eviction trick (browser behavior inconsistent across Safari/Chrome — not worth complexity).

- **D-13: HTMX forms in `dashboard.py` keep their existing `hx-headers='{"X-Trading-Signals-Auth": "{{WEB_AUTH_SECRET}}"}'` attribute UNCHANGED.** Zero diff to `dashboard.py` (1010..1099 form rendering, 1515..1565 partial-refresh trigger), zero diff to `web/routes/dashboard.py` placeholder substitution (Phase 14 Plan 14-04 Task 5 lines 73..76, 195..200) and its threat-T-14-15 mitigation tests in `tests/test_web_dashboard.py::TestAuthSecretPlaceholderSubstitution`. Belt-and-suspenders: HTMX form POST authenticates via the explicit header AND via the same-origin auto-sent cookie. If the cookie expires mid-day during a trade, the explicit header still authenticates the form — no 'sorry try again' confusion. AUTH-05 (additive contract) preserved verbatim. No CSRF token added (SameSite=Strict cookie + single-form-per-origin = no CSRF surface for a single-operator tool).

- **D-14: `GET /login` form structure** (planner's discretion on visual aesthetic, but contract-locked):
  - Two-field form: username (text) + secret (password)
  - Submit button: "Sign in"
  - On submit failure → re-render same form with single error message: "Sign in failed" (NO username-vs-password distinction per AUTH-02 spirit — "no leaked info; no hints"; same message + same response timing whether wrong username or wrong password)
  - On submit success → 302 to `?next=<path>` if `next` query param was set on incoming /login request (defaults to `/`)
  - Success path validates `next` is a relative path (starts with `/`, no `://`, no `.<scheme>:`) before redirecting — open-redirect prevention
  - Login form CSS: planner's discretion; mirror `dashboard.py::_INLINE_CSS` palette + typography for visual consistency. Shared CSS extraction is NOT required for this phase.

- **D-15: Folded Todo: `2026-04-27-phone-friendly-auth-ux-for-dashboard-access.md` (score 0.9).** Todo's "Problem" section becomes the source-of-truth narrative for AUTH-04 motivation. Todo's 4 candidate solutions: Options 1 + 2 fold into D-01 (this phase ships both); Options 3 + 4 fold into deferred (D-02). Todo's "Constraints to preserve" section maps directly to AUTH-05/AUTH-06/AUTH-07 (already in REQUIREMENTS.md per 2026-04-27 commit `cb33ff8`).

### Claude's Discretion

- **Plan-level wave structure inside 16.1-02.** Plan 16.1-02 (cookie wave) likely needs sub-tasks: itsdangerous serializer setup, GET /login handler, POST /login handler, POST /logout handler, AuthMiddleware cookie-validation extension, login form HTML template, /login + /logout tests. Planner sequences these.
- **Login form HTML template location.** Inline in `web/routes/login.py` (returned from handler as Jinja string) vs separate `web/templates/login.html` (introduces Jinja2 dependency if not already present — check `requirements.txt`). Planner picks; recommend inline if no other Jinja templates exist, or new `web/templates/` dir if Jinja2 is already pulled in.
- **`web/middleware/auth.py` refactor depth.** D-07's 4-step sniff order may justify extracting helpers (`_try_basic`, `_try_cookie`, `_try_header`, `_branch_unauthenticated`). Planner decides whether to refactor in 16.1-01, 16.1-02, or piecemeal.
- **Sec-Fetch header sniff fallback.** Modern browsers (Safari 14+, Chrome 76+, Firefox 90+) always send `Sec-Fetch-*` headers on navigations. Older browsers / non-browser HTTP clients don't. Planner: also fall back to `Accept: text/html` substring match for older/edge cases. If both signals miss, default to 401 plain-text (script-shaped).
- **Test file naming.** Recommend `tests/test_web_auth_basic.py` (Plan 16.1-01) and `tests/test_web_auth_cookie.py` + `tests/test_web_routes_login.py` (Plan 16.1-02), mirroring Phase 13's `tests/test_web_auth_middleware.py` naming.
- **`tests/test_web_auth_middleware.py` regression coverage.** Existing Phase 13 tests must keep passing unchanged (header path AUTH-05 preserved). Extend with new test classes — no rewrites of existing tests.
- **Whether to surface a "Sign out" button in `dashboard.py`.** UI nicety — small button in dashboard header / footer that POSTs to /logout. Planner discretion; recommend YES (otherwise the only logout path is typing `/logout` into the URL bar, which is awful UX). Renders only when a cookie session is detected (otherwise: no button, since header/Basic users can't meaningfully "log out" server-side).

Same exemption mechanism as `EXEMPT_PATHS` per Phase 13 D-02 but with awareness: `EXEMPT_PATHS_PUBLIC = frozenset({'/healthz', '/login'})` for GET-method-only public access; POST /login is rate-limited but publicly accessible (auth bootstrap); POST /logout is auth-required (only authenticated users can log out).
- `dashboard.py::render_dashboard` — extended to accept an optional `is_cookie_session: bool = False` parameter. When True, renders a "Sign out" button (form POSTing to /logout). When False (header-only or Basic Auth via URL bar), no button rendered. Caller (`web/routes/dashboard.py::get_dashboard`) detects cookie session by inspecting `request.cookies.get('tsi_session')` AND validating it (reuses middleware helper).
