### URLSafeTimedSerializer API contract (D-10)

```python
from itsdangerous.url_safe import URLSafeTimedSerializer

# Constructor — secret_key accepts str | bytes | Iterable[str] | Iterable[bytes]
serializer = URLSafeTimedSerializer(
  secret_key=os.environ['WEB_AUTH_SECRET'],   # str OK; library handles encoding
  salt='tsi-session-cookie',                   # bytes OR str; namespace per cookie
)

# Sign — returns URL-safe ASCII string (alphanumerics + . _ -)
token = serializer.dumps({'u': username, 'iat': int(time.time())})

# Verify — raises on invalid signature OR expired (>max_age seconds)
try:
  payload = serializer.loads(token, max_age=43200)  # 12h per D-11
except SignatureExpired:
  ...   # cookie aged out → fall through to D-07 step 4 (302 / 401 branch)
except BadSignature:
  ...   # tampered or wrong salt → same fall-through
```

### Exception hierarchy (verified against current docs)

```
BadData (root)
  └── BadSignature
        └── BadTimeSignature
              └── SignatureExpired
```

`[CITED: itsdangerous TimedSerializer docs https://itsdangerous.palletsprojects.com/en/stable/timed/]`

**Implication for D-07 step 2:** A single `except BadSignature:` clause in `_try_cookie` catches BOTH expired AND tampered cookies. If the audit log needs to distinguish them (Claude's Discretion — recommend YES so log lines say `reason=expired` vs `reason=bad_sig`), use:
```python
except SignatureExpired:
  self._log_failure(request, reason='cookie_expired')
except BadSignature:
  self._log_failure(request, reason='cookie_bad_sig')
```
The narrower `except SignatureExpired` MUST come first (stdlib LEGB rule).

### Salt parameter — non-optional per Phase 16.1

The `salt=` kwarg is technically optional (defaults to `b'itsdangerous'`), but using a per-purpose salt is best practice — it ensures a token signed for one purpose cannot be replayed against a different `URLSafeTimedSerializer` reusing the same secret. Recommend `salt='tsi-session-cookie'` (matches the cookie name `tsi_session` for grep-ability).

## Sec-Fetch-* sniff implementation

### Browser support matrix — CRITICAL CORRECTION TO CONTEXT.md D-04

CONTEXT.md §Specific Ideas (line 193) claims "Sec-Fetch-Mode was added in ... Safari 14.1 (Apr 2021)". **This is wrong.** Per `[VERIFIED: github.com/mdn/browser-compat-data/issues/19389 + Apple Safari 16.4 release notes]`:

| Browser | Min version supporting Sec-Fetch-* | Year |
|---------|------------------------------------|------|
| Chrome (desktop + Android) | 76 | Aug 2019 |
| Firefox | 90 | Jul 2021 |
| Safari (macOS) | **16.4** | Mar 2023 |
| Safari (iOS / iPadOS) | **16.4** | Mar 2023 |

**Operator impact:** if the operator's iPhone runs iOS < 16.4, Sec-Fetch headers will NOT be sent. The fallback heuristic (`'text/html' in Accept` substring match) is therefore LOAD-BEARING, not just a defensive belt-and-suspenders. The CONTEXT.md Claude's Discretion §"Sec-Fetch sniff fallback" assumes the fallback is for "older/edge cases" — for iOS 14, 15, and 16.0–16.3, the fallback IS the primary detection.

**Recommended planner action:**
1. Verify with operator: which iOS version is the iPhone running? If iOS ≥ 16.4 → Sec-Fetch primary, Accept fallback. If iOS < 16.4 → Accept fallback IS primary.
2. Either way, the implementation order should be: try Sec-Fetch headers first (more specific); if absent, fall back to `Accept` substring; if neither matches → script-shaped → 401. This is robust to either iOS version.

### Definitive "is browser navigation" check

`[CITED: MDN Sec-Fetch-Mode + Sec-Fetch-Dest reference]`

A real browser navigation (address-bar paste, link click, form GET-action) sends ALL FOUR:
```
Sec-Fetch-Dest: document
Sec-Fetch-Mode: navigate
Sec-Fetch-Site: same-origin   (or none / cross-site depending on referrer)
Sec-Fetch-User: ?1            (only when user-initiated)
```

**Most specific 2-condition check** (resilient against XHR mode-spoofing):
```python
def _is_browser_navigation(request) -> bool:
  '''Returns True for actual browser top-level navigation.'''
  mode = request.headers.get('sec-fetch-mode', '').lower()
  dest = request.headers.get('sec-fetch-dest', '').lower()
  if mode == 'navigate' and dest == 'document':
    return True
  # Fallback for browsers < Sec-Fetch support (iOS Safari < 16.4):
  # Accept header is the only signal. Use substring match — Safari sends
  # 'text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8'
  # and curl WITHOUT --header sends '*/*' (no 'text/html').
  accept = request.headers.get('accept', '').lower()
  return 'text/html' in accept and 'sec-fetch-mode' not in {h.lower() for h in request.headers}
```

The second-clause guard (`'sec-fetch-mode' not in headers`) is important: it ensures the Accept fallback only triggers when Sec-Fetch is ABSENT. Otherwise an XHR with `Accept: text/html` (rare but possible) would falsely match. With the guard, an XHR sends `Sec-Fetch-Mode: cors`, the Sec-Fetch path runs first (returns False), and the Accept path is skipped.

### HTMX request Sec-Fetch values — D-13 belt-and-suspenders verification

`[VERIFIED: MDN + http.dev]` HTMX uses `XMLHttpRequest` under the hood, which sends:
```
Sec-Fetch-Mode: cors        (default for XHR)
Sec-Fetch-Dest: empty       (per MDN: fetch/AJAX/XHR/WebSocket → empty)
Sec-Fetch-Site: same-origin (HTMX request to same origin)
```

**Implication:** the `_is_browser_navigation` check above returns False for HTMX requests (mode=cors, not navigate). So an HTMX form POST without a valid auth header would NOT redirect to /login — it would 401 plain-text. This is the correct behavior per D-05: HTMX is "script-shaped" and should get the script contract.

**But:** Phase 14 HTMX forms carry the `X-Trading-Signals-Auth` header explicitly per D-13 — they always pass D-07 step 3 (header validation). The 401 path is only hit if the header is missing or wrong, which would be a legitimate HTMX failure case (e.g., env-var rotation mid-session). Per D-13 belt-and-suspenders, the cookie also authenticates the same request — so even if the header validation fails (e.g., placeholder substitution glitch), the cookie validation in step 2 would still grant. **This is the architectural insight that makes D-13 robust.**

### `Accept: text/html` substring fallback safety

**False-positive scrutiny:** can a non-browser request reasonably set `Accept: text/html`?

| Client | Default `Accept` header | False-positive risk? |
|--------|-------------------------|---------------------|
| `curl` (no `-H`) | `*/*` | None |
| `curl -H "Accept: text/html"` | `text/html` | YES — operator could intentionally trigger 302 to /login. Acceptable: explicit opt-in to browser path |
| Python `requests` (default) | `*/*` | None |
| `wget` | `*/*` | None |
| HTMX XHR | Whatever the form/request sets, often `*/*` or `application/json` | None (also blocked by Sec-Fetch-Mode: cors guard above) |
| Browser navigation | `text/html,application/xhtml+xml,...` | TRUE positive — desired |
| Browser XHR / fetch() | `*/*` (default) or developer-set | None (blocked by Sec-Fetch-Mode: cors guard) |

**Conclusion:** the `Accept: text/html` substring check, combined with the "Sec-Fetch absent" guard, has ZERO false positives in practice. The only ways to trigger it are: (a) a real browser navigation, or (b) deliberate `curl --header "Accept: text/html"` — which is an explicit operator choice, not a script accident.

## Starlette middleware chaining (4-step sniff per D-07)

### Recommended pattern: extend BaseHTTPMiddleware in place

`[CITED: github.com/Kludex/starlette/discussions/2160]` BaseHTTPMiddleware is NOT being deprecated despite a 2023 proposal — Starlette maintainers decided to keep it but discourage NEW use through the documentation's "Limitations" section. **For Phase 16.1, extending the existing `AuthMiddleware(BaseHTTPMiddleware)` is the correct choice** because:

1. The existing class is already BaseHTTPMiddleware (Phase 13 D-01).
2. The known limitations (contextvar propagation, body-buffer interactions, exception swallowing) all apply ONLY when calling `await call_next(request)`. **Early returns (302/401) are unaffected** per `[VERIFIED: github.com/Kludex/starlette/discussions/1729]`.
3. Phase 13's docstring already flags the contextvar limitation for Phase 14+; Phase 16.1 does NOT introduce contextvars or BackgroundTasks. The warning remains accurate without amendment.

**Do NOT migrate to pure ASGI middleware in Phase 16.1.** That's a v1.2+ refactor if Phase 14's BackgroundTask warning ever materializes.

### Mid-dispatch early-return pattern (4 conditions)

```python
async def dispatch(self, request, call_next):
  # D-02: exempt allowlist FIRST — never reach auth logic for /healthz
  if request.url.path in EXEMPT_PATHS:
    return await call_next(request)

  # D-07 step 1: Basic Auth
  if self._try_basic(request):
    return await call_next(request)

  # D-07 step 2: signed cookie
  if self._try_cookie(request):
    return await call_next(request)

  # D-07 step 3: header (Phase 13 path, unchanged)
  if self._try_header(request):
    return await call_next(request)

  # D-07 step 4: log + branch
  self._log_failure(request)
  if self._is_browser_navigation(request):
    next_path = quote(request.url.path)
    if request.url.query:
      next_path += '?' + quote(request.url.query)
    return Response(
      status_code=302,
      headers={'Location': f'/login?next={next_path}'},
    )
  return Response(
    content='unauthorized',
    status_code=401,
    media_type='text/plain; charset=utf-8',
  )
```

**Critical detail:** `_try_*` helpers return `bool` (granted vs not-granted), NOT a tuple or response. This keeps each check independent and lets the dispatch driver decide the failure response shape ONCE, in step 4. Helpers must NOT call `_log_failure` themselves — only step 4 logs (otherwise a request that fails step 1, succeeds step 2 would log a spurious failure).

### Constructing 302 / 401 responses inline

Both `starlette.responses.Response` and `starlette.responses.RedirectResponse` work for the redirect, but `Response(status_code=302, headers={'Location': ...})` is more explicit (RedirectResponse defaults to 307 which preserves the request method — wrong for GET-after-failed-POST). **Use plain `Response` with explicit `status_code=302`.**

For the `?next=` parameter URL-encoding, use `urllib.parse.quote` with default `safe='/'` — preserves slashes (necessary for relative paths) but encodes everything else (spaces, query strings).

Phase 13's `web/middleware/auth.py:23-27` warns:

## HTTP Basic Auth implementation

### Decode pattern (avoid `split(':', 1)` pitfall)

```python
import base64
import binascii
import hmac

def _try_basic(self, request) -> bool:
  '''D-07 step 1: validate Authorization: Basic <b64> header.'''
  auth = request.headers.get('authorization', '')
  if not auth.lower().startswith('basic '):
    return False
  try:
    decoded = base64.b64decode(auth[6:].strip(), validate=True).decode('utf-8')
  except (binascii.Error, UnicodeDecodeError):
    return False
  # CRITICAL: split on FIRST colon only. partition() is unambiguous about
  # what happens when no colon is present (returns (whole_string, '', '')),
  # whereas split(':', 1) returns a 1-element list which is harder to
  # destructure safely.
  username, sep, password = decoded.partition(':')
  if not sep:
    return False  # no colon → malformed
  # D-03 reuse: hmac.compare_digest for BOTH fields (D-08 + AUTH-06).
  username_match = hmac.compare_digest(
    username.encode('utf-8'), self._username_bytes,
  )
  password_match = hmac.compare_digest(
    password.encode('utf-8'), self._secret_bytes,
  )
  return username_match and password_match
```

### Why `partition(':')` over `split(':', 1)`

| Input | `split(':', 1)` | `partition(':')` |
|-------|----------------|------------------|
| `'user:pw'` | `['user', 'pw']` (2 elements) | `('user', ':', 'pw')` |
| `'no_colon'` | `['no_colon']` (1 element) | `('no_colon', '', '')` |
| `':just_password'` | `['', 'just_password']` | `('', ':', 'just_password')` |
| `'user:pw:with:colons'` | `['user', 'pw:with:colons']` | `('user', ':', 'pw:with:colons')` |

Both work for the well-formed case. `partition` makes the malformed case (no colon) trivially detectable via `if not sep` — `split` requires `if len(parts) != 2` which is equivalent but less self-documenting.

**Why `validate=True` on `b64decode`:** `[CITED: Python stdlib base64 docs]` rejects whitespace and non-alphabet characters that are silently ignored by default. A malformed header should fail fast → return False → fall through to next sniff step.

### Behavior without WWW-Authenticate challenge — verified

`[VERIFIED: HTTP RFC 7235 §2.1 + Safari WebKit issue #37989]` Browsers WILL send an `Authorization: Basic <b64>` header on a request if:

1. The user types `https://user:password@host/path` in the address bar (URL-bar form), OR
2. The credentials are in the browser's keychain AND the browser decides to re-send them preemptively, OR
3. JavaScript explicitly sets the header via `fetch({headers: {Authorization: ...}})`.

**No WWW-Authenticate header is required server-side** for any of these. The system Basic-Auth dialog (which IS triggered by WWW-Authenticate: Basic) is the ONLY browser behavior that requires the server to send the challenge.

**Conclusion:** D-04..D-05 ("never send WWW-Authenticate") is implementable without breaking the URL-bar Basic Auth path.

### iOS Keychain caching of Basic Auth — MEDIUM confidence

`[CITED: Apple Discussions thread 8414341, 3274071 + Daniel Schneller blog post 2018]`

**Caveat:** iOS Safari's Keychain handling of Basic Auth credentials is INCONSISTENT:
- iOS stores Basic Auth credentials as "Web form password" (not "Internet Password" like macOS Safari does).
- iOS Safari does NOT auto-fill Basic Auth credentials from the keyboard toolbar key icon — credentials must be typed manually OR the user re-uses the saved `https://user:pw@host/` URL pattern.
- After typing `https://user:pw@signals.<owned-domain>.com/` once, Safari MAY save the URL in History/Autocomplete (URL-bar suggestion on next visit). This is the actual UX path — not "Keychain auto-fill", but "address-bar history".
- iOS Safari preserves the Authorization header for subsequent same-origin requests within the same session/tab — `[CITED: Safari issue: Authorization header sent on every request after first Basic Auth]`. This is what makes the URL-bar form usable for the operator: type credentials once, all subsequent navigation reuses the cached header.

**Operator UX expectation:** "type the URL with creds once per session, dashboard works for the rest of the tab's lifetime, close tab → next visit needs creds again". This matches D-12's logout-page footnote ("close all Safari tabs to fully clear them").

## Open-redirect prevention (`?next=`)

### Known bypass payloads to block

`[CITED: diverto.hr 2024-12 open-redirect bypass post + claroty.com URL parsing confusion]`

| Payload | Why it bypasses naive `startswith('/')` checks |
|---------|------------------------------------------------|
| `//evil.com` | `startswith('/')` is True; protocol-relative URL → browser navigates to `https://evil.com` |
| `///evil.com/path` | Same — leading slashes count as protocol-relative |
| `/\evil.com` | WHATWG URL spec treats `\` as `/`; modern browsers normalize |
| `\\evil.com` | Same backslash issue; some `urlparse` implementations differ from browsers |
| `\/evil.com` | Same |
| `%2f%2fevil.com` | URL-encoded `//`; some servers decode before validation |
| `%2F\evil.com` | Mixed encoding + backslash |
| `javascript:alert(1)` | `startswith('/')` is False but if redirect uses `Location:` header AND server tolerates non-relative, bad clients render it |
| `data:text/html,<script>...` | Same as javascript: |
| `https://signals.<owned-domain>.com.evil.com/` | Domain confusion — appears to be on trusted host |
| `https://signals.<owned-domain>.com@evil.com/` | URL userinfo @ trick — browser navigates to evil.com |

### Defensible one-line validator

```python
def _is_safe_next(next_value: str) -> bool:
  '''D-14 open-redirect prevention. Returns True iff `next_value` is a
  same-origin relative path safe to redirect to.

  Defenses:
    1. Must start with '/' (relative path)
    2. Must NOT start with '//' or '/\\' (protocol-relative)
    3. Must NOT contain '://' anywhere (absolute URL)
    4. Must NOT contain backslash (WHATWG normalizes \\ → /)
    5. Must NOT contain control chars or whitespace (CR/LF, %0a, etc.)
    6. Length cap to prevent log-flooding via giant ?next=

  All other validation (path traversal '../') is the route handler's
  problem — the redirect target is FastAPI's own URL space, not a
  filesystem path.
  '''
  if not next_value or not isinstance(next_value, str):
    return False
  if len(next_value) > 512:  # generous cap; reasonable URLs are <200 chars
    return False
  if not next_value.startswith('/'):
    return False
  if next_value.startswith('//') or next_value.startswith('/\\'):
    return False
  if '\\' in next_value:  # WHATWG-equivalence guard
    return False
  if '://' in next_value:  # absolute URL guard (catches e.g. /a/b://c)
    return False
  # Control char + whitespace guard (CR/LF response splitting, tabs, etc.)
  if any(ord(c) < 0x20 or c == '\x7f' for c in next_value):
    return False
  return True

# Usage in POST /login success path:
next_value = request.query_params.get('next', '/')
target = next_value if _is_safe_next(next_value) else '/'
return Response(status_code=302, headers={'Location': target})
```

**Note on URL-encoding bypasses:** Starlette / FastAPI decode `request.query_params['next']` automatically — `%2F%2Fevil.com` becomes `//evil.com` BEFORE `_is_safe_next` runs. So the percent-decoded form is what gets validated. The `startswith('//')` guard catches it. **Do NOT manually decode again** (would create double-decode bypass: `%252F` → `%2F` first decode → `/` second decode = bypass).

**Why we don't use `urllib.parse.urlparse` here:** Python's `urlparse` is RFC 3986-conformant; modern browsers follow WHATWG which differs on backslash normalization (`[CITED: hacktricks book + python urlparse limitations]`). A `urlparse('//evil.com').netloc` returns `'evil.com'` (correctly identifies as protocol-relative), but `urlparse('/\evil.com').netloc` returns `''` and `.path` returns `/\evil.com` — `urlparse` thinks it's relative, browser thinks it's `/evil.com`. Hand-rolled string checks above are MORE conservative than `urlparse` for this specific use case.

## Cookie hygiene specifics

### TestClient + Secure cookie behavior

`[VERIFIED: github.com/fastapi/fastapi/issues/3339]` FastAPI/Starlette TestClient **drops Secure cookies on HTTP test requests** (it follows browser-like rules: Secure cookies require HTTPS). The TestClient base URL defaults to `http://testserver`, so any `Set-Cookie: ...; Secure` header issued by the app will be present on the response BUT will not be sent back on subsequent client requests.

**Implication for D-15 cookie tests:**

Don't rely on round-trip cookie behavior in TestClient. Instead, use a two-pronged test strategy:

```python
def test_login_sets_secure_session_cookie(client):
  '''D-11/D-12: assert Set-Cookie header text directly (round-trip not testable).'''
  response = client.post('/login', data={
    'username': 'marc',
    'secret': 'a' * 32,
  }, follow_redirects=False)

  # Inspect raw header text — TestClient drops Secure cookies from response.cookies
  set_cookie = response.headers.get('set-cookie', '')
  assert 'tsi_session=' in set_cookie
  assert 'Max-Age=43200' in set_cookie
  assert 'Path=/' in set_cookie
  assert 'Secure' in set_cookie
  assert 'HttpOnly' in set_cookie
  assert 'SameSite=Strict' in set_cookie

def test_cookie_round_trip_via_explicit_set(client, valid_cookie_token):
  '''Cookie validation works end-to-end when the cookie is INJECTED manually
  (bypasses TestClient's Secure-drop behavior).'''
  response = client.get('/', cookies={'tsi_session': valid_cookie_token})
  assert response.status_code == 200
```

**The `valid_cookie_token` fixture** uses the SAME `URLSafeTimedSerializer` instance that the production middleware uses, with the SAME secret + salt — so the cookie is signature-valid. Token construction lives in `tests/conftest.py` next to `VALID_SECRET`.

### SameSite=Strict implications for /login POST

`[CITED: MDN SameSite cookies]` SameSite=Strict means:
- The cookie is sent on same-origin requests (POST /login submits to same origin → ✓)
- The cookie is NOT sent on cross-site navigations — even if the user clicks a link from another site to `signals.<owned-domain>.com/dashboard`, the cookie is suppressed on that initial navigation. The user sees the login form on that one navigation; subsequent same-origin clicks send the cookie.

**Operator UX impact:** if the operator bookmarks `signals.<owned-domain>.com/` and clicks the bookmark from a search-results page, the cookie may not be sent on the bookmarked-link navigation. They'll see /login. Same-origin reload sends the cookie. **This is acceptable** for a single-operator tool but worth documenting in the deferred-ideas section.

### Cookie deletion attrs (D-12 — global LEARNINGS direct application)

```python
# Creation:
Set-Cookie: tsi_session=<token>; Max-Age=43200; Path=/; Secure; HttpOnly; SameSite=Strict

# Deletion (must MATCH all attrs except Max-Age):
Set-Cookie: tsi_session=; Max-Age=0; Path=/; Secure; HttpOnly; SameSite=Strict
```

Verification command in PLAN.md:
```python
def test_logout_deletion_attrs_match_creation(client):
  # Login first
  ...
  # Logout
  response = client.post('/logout', cookies={'tsi_session': token})
  set_cookie = response.headers.get('set-cookie', '')
  for attr in ('tsi_session=', 'Max-Age=0', 'Path=/', 'Secure', 'HttpOnly', 'SameSite=Strict'):
    assert attr in set_cookie, f'missing attr in deletion: {attr!r}'
```

If even one attr is missing, the browser silently keeps the original cookie. The test catches this on every CI run.

### Sampling Pyramids

Two sparse-signal areas where the executor should add explicit assertions to avoid silent failures:

**Pyramid 1 — Audit-log fires on EACH new auth path failure**

The `_log_failure` method in `web/middleware/auth.py` is called from D-07 step 4. It must fire on:
- Failed Basic Auth (wrong username, wrong password, malformed header)
- Failed cookie validation (expired, tampered, malformed)
- Failed header validation (unchanged Phase 13 behavior)
- Combination misses (e.g., wrong Basic AND wrong header on same request → log ONCE per request, not 3x)

```python
@pytest.mark.parametrize('headers,cookies', [
  ({'Authorization': 'Basic d3JvbmdfdXNlcjp3cm9uZ19wdw=='}, {}),  # bad Basic only

])

  client.get('/', headers=headers, cookies=cookies)

```

**Pyramid 2 — Cookie payload re-signs on logout / re-login (no token reuse after logout)**

If the cookie is stateless (no server-side revocation list), then a stolen cookie remains valid until its 12h TTL expires — even after the operator logs out. This is the architectural reality per CONTEXT §Deferred Ideas "Cookie session storage on server".

**Sampling test:** explicitly assert the threat-model boundary so the operator's mental model matches the code:

```python
def test_logout_does_NOT_invalidate_existing_cookie_value(client):
  '''ARCHITECTURAL: stateless cookies cannot be revoked server-side.
  This test documents (does not fix) the limitation. If a future phase
  adds a revocation list (Redis / on-disk), update this test to assert
  the OPPOSITE behavior, and remove the deferred-idea entry.
  '''
  # Login → grab cookie value
  login_response = client.post('/login', data={'username': 'marc', 'secret': 'a' * 32})
  cookie_value = parse_set_cookie(login_response.headers['set-cookie'])['tsi_session']
  # Logout
  client.post('/logout', cookies={'tsi_session': cookie_value})
  # Same cookie still works (THIS IS THE LIMITATION)
  reuse_response = client.get('/', cookies={'tsi_session': cookie_value})
  assert reuse_response.status_code == 200, 'Cookie revocation NOT in scope for v1.1'
```

This counter-intuitive assertion documents the design choice in code. If a v1.2 phase adds revocation, this test inverts to `assert reuse_response.status_code == 401`.

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
