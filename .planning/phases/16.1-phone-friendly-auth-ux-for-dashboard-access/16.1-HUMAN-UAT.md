# Phase 16.1 — HUMAN-UAT (Operator Verification)

> Seven scenarios covering the new cookie+TOTP+trusted-device+magic-link auth UX shipped in Phase 16.1 (Plans 16.1-01, 16.1-02, 16.1-03). Operator runs each scenario on a real iPhone (Safari preferred; Chrome accepted) plus a Mac terminal for curl regression rows, fills in the 5 fields per Phase 16 D-10, and the next /gsd-verify-work invocation mirrors the verified-status rows into STATE.md §Completed Items.

**D-10 schema (5 fields per scenario):**
1. **Scenario ID** — stable identifier
2. **Original scenario** — N/A for new-for-16.1 scenarios; path link otherwise
3. **Verification status** — `pending` / `verified` / `partial`
4. **Operator verification date** — ISO `YYYY-MM-DD` once verified
5. **Operator notes** — free text, screenshot path, issues observed

**Pre-flight:** before running scenarios, deploy 16.1 to the droplet (`bash deploy.sh`) and confirm `https://signals.<owned-domain>.com/healthz` returns `HTTP/2 200` over the live TLS chain. Open the operator's authenticator app of choice (Google Authenticator, 1Password, Authy) so QR scans during enrollment scenarios succeed.

## UAT-16.1-A: iPhone Safari first-visit + cookie login + TOTP enrollment

**Scenario ID:** UAT-16.1-A
**Original scenario:** N/A — new for Phase 16.1
**Verification status:** pending
**Operator verification date:**
**Operator notes:**

**How to verify:**
1. On iPhone Safari, open `https://signals.<owned-domain>.com/`. Expect 302 → `/login` and the sign-in form to render. Confirm:
   - Title "Trading Signals — Sign in"
   - Two inputs labelled `Username` and `Password` (UI-SPEC §Surface 1 palette)
   - Link "Lost 2FA? Reset via email" beneath the submit button
   - No browser Basic Auth dialog (per LEARNING `WWW-Authenticate` is never sent)
2. Type `WEB_AUTH_USERNAME` (e.g. `marc`) and `WEB_AUTH_SECRET` (32+ chars). Submit.
3. Expect 302 → `/enroll-totp` with QR code rendered, manual-entry secret displayed, and 6-digit code input.
4. Open the authenticator app and scan the QR code. The app should add an entry "Trading Signals (mwiriadi@signals.<owned-domain>)".
5. Type the 6-digit code from the authenticator app into the form. Submit.
6. Expect 302 → `/` (dashboard) with `Set-Cookie: tsi_session=...` issued.
7. Confirm the dashboard renders fully:
   - Signal cards (SPI200, AUDUSD)
   - Equity chart
   - Header has "Sign out" button (cookie-session indicator)
8. Tap "Sign out". Expect 200 with "Signed out" confirmation page; tsi_session cleared (Max-Age=0 + matching deletion attrs per LEARNING).
9. Update **Verification status** to `verified` if all steps pass; `partial` with notes if any UI element is misaligned.

> **Note:** this scenario cannot be FULLY verified within a single session — full 30-day persistence requires a follow-up check 1+ days later. Operator may legitimately leave this `partial` while the immediate skip works (D-17 escape hatch).

**How to verify:**
1. Repeat the UAT-16.1-B flow (close tabs, sign in, reach `/verify-totp`).
2. At `/verify-totp`, type the 6-digit code AND check the "Trust this device for 30 days" box. Submit.

4. Close all Safari tabs, then reopen `https://signals.<owned-domain>.com/`.
5. **Immediate skip path:** expect 302 → `/login`, then enter creds → expect 302 → `/` DIRECTLY (no `/verify-totp` prompt — trusted device cookie is presented and accepted alongside the post-login state).

6. **Long-persistence path:** wait 1+ days (or use the operator's tolerance for this). Repeat steps 4-5; expect the same skip behaviour. Update notes when verified or downgrade if the skip stops working before 30 days.
7. Update **Verification status** to `verified` once both immediate AND multi-day skip are observed; `partial` if only the immediate skip is verified within this session.

## UAT-16.1-D: Lost-phone scenario + magic-link reset (E-07 happy path)

**Scenario ID:** UAT-16.1-D
**Original scenario:** N/A — new for Phase 16.1
**Verification status:** pending
**Operator verification date:**
**Operator notes:**

**How to verify:**
1. On iPhone Safari, open `https://signals.<owned-domain>.com/login`.
2. Tap the "Lost 2FA? Reset via email" link beneath the submit button.
3. Expect 200 + `/forgot-2fa` form: title "Reset 2FA", username + password inputs, "Send reset link" button.
4. Enter the same `WEB_AUTH_USERNAME` + `WEB_AUTH_SECRET`. Submit.
5. Expect 200 + "Check your email" page (single static copy regardless of cred validity per E-07).
6. Open Gmail (operator's `OPERATOR_RECOVERY_EMAIL` inbox = `mwiriadi@gmail.com` by default). Within ~30 seconds, an email should arrive:
   - From: `signals@carbonbookkeeping.com.au` (or whatever `SIGNALS_EMAIL_FROM` is set to)
   - Subject (verbatim): `Trading Signals — 2FA reset link (valid 1 hour)`
   - Body: prominent "Reset 2FA" button + plain-text fallback link + AWST-formatted expiry timestamp
7. Tap "Reset 2FA" in the email. Expect 302 → `/enroll-totp?reset=1` with a new `tsi_session` cookie set + Referrer-Policy: no-referrer header.
8. Expect the reset-mode page: TWO buttons — "Keep current authenticator" and "Set up new authenticator".
9. **Path A — Keep:** tap "Keep current authenticator". Expect 302 → `/`. The original authenticator entry still works.
10. **Path B — New (recommended for "lost phone" — UAT covers this path):** tap "Set up new authenticator".
    - Expect 200 with a fresh QR code + new manual-entry secret.
    - In the authenticator app on the NEW phone, scan the new QR. Old entry will be stale (expected).
    - Type the 6-digit code from the new entry. Submit.
    - Expect 302 → `/` with `tsi_session` set; dashboard renders.
11. **Replay test:** revisit the magic-link URL from the email. Expect 401 + "Reset link is no longer valid" page (token already consumed — T-16.1-24).
12. Update **Verification status**.

> **Failure modes to watch for and report:**
> - Email never arrives in Gmail: check `journalctl -u trading-signals.service` on the droplet for `[Email]` log lines; confirm `SIGNALS_EMAIL_FROM` and `RESEND_API_KEY` are set in the systemd env and that the operator's recovery email matches `OPERATOR_RECOVERY_EMAIL`.
> - Email arrives but link is `localhost`: indicates `BASE_URL` env var is missing on the droplet — service should have logged `[Web] BASE_URL env var not set — magic-link email skipped` and email should NOT have been sent. Investigate.
> - Magic-link URL clicks succeed but `/enroll-totp?reset=1` shows the standard verify form instead of Keep/New buttons: check that Phase 16.1 Plan 03 deployment landed (grep `_render_enroll_reset_choice_page` in deployed `web/routes/totp.py`).

## UAT-16.1-F: AUTH-05 regression — curl with header still works

**Scenario ID:** UAT-16.1-F
**Original scenario:** Phase 13 AUTH-05 (`X-Trading-Signals-Auth` header path)
**Verification status:** pending
**Operator verification date:**
**Operator notes:**

**How to verify:**
1. From Mac terminal:
   ```bash
   curl -i -H "X-Trading-Signals-Auth: <secret>" https://signals.<owned-domain>.com/
   ```
   where `<secret>` = the same `WEB_AUTH_SECRET` set in the droplet's `.env`.
2. Expect:
   - `HTTP/2 200`
   - `content-type: text/html; charset=utf-8`
   - Body is the dashboard HTML (full ~20KB render — same shape as UAT-16-A from Phase 16)

3. Repeat for `/api/state`:
   ```bash

   ```
   Expect HTTP/2 200 + JSON body.
4. Update **Verification status**.

> **Failure note:** if any of these curl calls now return 302 or 401, the new cookie+TOTP UX has accidentally broken the legacy header path — block phase closure until restored.

## UAT-16.1-G: AUTH-07 regression — curl without auth still 401

**Scenario ID:** UAT-16.1-G
**Original scenario:** Phase 13 AUTH-07 (401 plain-text on no-auth, no `WWW-Authenticate`)
**Verification status:** pending
**Operator verification date:**
**Operator notes:**

**How to verify:**
1. From Mac terminal, no Sec-Fetch headers (curl default), no `Accept: text/html`:
   ```bash
   curl -i https://signals.<owned-domain>.com/
   ```
2. Expect:
   - `HTTP/2 401`
   - `content-type: text/plain; charset=utf-8`
   - Body literal: `unauthorized`
   - **NO** `Set-Cookie` header
   - **NO** `Location` header (no 302 — script callers stay 401)
   - **NO** `WWW-Authenticate` header (per LEARNING — Phase 13 D-04 + Phase 16.1 D-04..D-07 reconciliation)
3. Repeat with explicit text/html Accept (curl simulates a browser navigation):
   ```bash
   curl -i -H "Accept: text/html" https://signals.<owned-domain>.com/
   ```
   Expect 302 + `Location: /login?next=%2F` (browser-shape per E-02 Sec-Fetch fallback to Accept).
4. Both behaviours observed → contract intact. Update **Verification status**.

> **Failure note:** if no-auth curl now returns a `WWW-Authenticate` header, Plan 16.1-01's middleware extension regressed — block phase closure.
