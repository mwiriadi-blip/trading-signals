# Phase 14 — UI Design Contract

> Visual and interaction contract. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Classification

**Extension phase — first phase to add new visual surface to the v1.0 dashboard.**

Phase 13 was a passthrough (server-side auth + read endpoints, no UI). Phase 14 modifies `dashboard.py::render_dashboard` to introduce three HTMX-driven surfaces:

1. **Open form** — one form rendered above the Open Positions table
2. **Close + Modify inline buttons** — per-row action affordance on each open position
3. **Inline error region** — per-form `<div class="error">` block that receives 400 field-level errors

All other dashboard surfaces (header, signal cards, equity chart, trades table, key stats, footer) are untouched — Phase 14 has no licence to modify them.

**Inheritance discipline.** The v1.0 design system in `dashboard.py::_INLINE_CSS` and `system_params._COLOR_*` is the sole source of truth for colour, type, spacing, and font tokens. Phase 14 ADDS new component patterns (forms, buttons, errors, manual-stop badge) that compose these existing tokens — never new tokens.

**v1.0 aesthetic constraints carried into Phase 14:**
- Dark dashboard (`_COLOR_BG = #0f1117`)
- Single-operator (no multi-user UX, no help text, no tooltips beyond what's strictly needed for safety)
- Terminal-like density (compact rows, monospace numbers via `var(--font-mono)`, no chrome bloat)
- HTMX (no React, no client framework, no build step)
- Auth gate is `X-Trading-Signals-Auth` header (Phase 13 D-01) — HTMX MUST attach this header to every POST via `hx-headers` on the form (or a body-level default)

## Locked Discretion Decisions

The seven items called out in the orchestrator brief as "user-visible decisions" — locked here as design contract. The planner and executor MUST implement these literally.

### Decision 1 — Open form placement

**Decision:** Single open form positioned at the **top of the Open Positions section, ABOVE the table**, after the `<h2 id="heading-positions">Open Positions</h2>` heading and before the `<table class="data-table">`.

**Rationale:** Aligns with v1.0 dashboard's "compose top-down" rhythm (eyebrow → action → data). Keeps the operator's action surface visible whether the table is full or empty (`— No open positions —`). One form for both instruments — `<select name="instrument">` toggles. No modal (modals don't fit the dense, terminal-like aesthetic and need extra JS).

**Markup contract (planner picks final IDs and class names but the structure is fixed):**

```html
<section class="open-form" hx-headers='{"X-Trading-Signals-Auth": "..."}'>
  <p class="eyebrow">OPEN NEW POSITION</p>
  <form
    hx-post="/trades/open"
    hx-target="#positions-tbody"
    hx-swap="innerHTML"
    hx-on::after-request="..."  <!-- handles 400 → inline error injection -->
  >
    <!-- 4 required fields (instrument, direction, entry_price, contracts) inline -->
    <!-- 4 optional fields inside <details><summary>Advanced</summary>...</details> -->
    <button type="submit" class="btn-primary">Open Position</button>
  </form>
  <div class="error" role="alert" aria-live="polite"></div>
  <!-- error region; empty until 400 returned -->
</section>
```

The `.open-form` section gets `margin-bottom: var(--space-6)` to separate it from the table. The form is rendered server-side by `dashboard.py` on every full render — not by HTMX. HTMX only handles the POST + response swap.

### Decision 2 — Close + Modify button placement

**Decision:** Inline action buttons in a **new last column** added to the Open Positions table, header `Actions`. Each button is text-only (no icon), rendered as `<button class="btn-row">Close</button>` and `<button class="btn-row">Modify</button>` separated by an 8px gap (`var(--space-2)`).

**Empty / FLAT positions:** When `state['positions'][instrument] is None`, the row is omitted entirely (existing v1.0 behaviour at `dashboard.py:696-697`) — there is no "ghost row" with greyed-out buttons. Operators open positions exclusively through the top-of-section open form.

**Markup contract:**

```html
<!-- New last column appended to existing 8 columns → 9 total -->
<thead>
  <tr>
    ...existing 8 <th> cells...
    <th scope="col">Actions</th>
  </tr>
</thead>
<tbody id="positions-tbody">
  <tr id="position-row-SPI200">
    ...existing 8 <td> cells...
    <td>
      <button class="btn-row btn-close"
```

**Note for planner:** The `hx-get` GET endpoints (`/trades/close-form`, `/trades/modify-form`) are planner discretion — they MAY be added as small server-rendered partials, OR the inline forms MAY be rendered server-side as hidden siblings revealed by JS. Recommended: server-rendered partial endpoints (no client JS). These GET endpoints are NOT in REQUIREMENTS.md but are HTMX glue — the planner adds them as supporting endpoints behind the auth gate.

**Empty-state colspan correction:** `dashboard.py:733` currently uses `colspan="8"`. Phase 14 changes the empty-state row to `colspan="9"` to span the new Actions column. This is the only modification to existing v1.0 markup outside the new column itself.

### Decision 3 — HTMX swap target and strategy

**Decision: per-row swap for close/modify; full-tbody swap for open.**

| Endpoint | `hx-target` | `hx-swap` | Server response shape |
|----------|-------------|-----------|------------------------|
| `POST /trades/open` | `#positions-tbody` | `innerHTML` | Server returns the full re-rendered `<tbody>` content (all visible position rows including the just-opened one). Avoids the "where do I insert this row?" race when a pyramid-up grows an existing row instead of adding a new one. |
| `POST /trades/close` | `#position-row-{instrument}` | `outerHTML` | Server returns either: (a) empty string + `HX-Trigger: positions-changed` response header → row vanishes from the DOM and a tbody-level listener re-renders the empty-state row when the last row goes; OR (b) the empty-state `<tr>` if this was the last row. Recommended: option (a) for simplicity. |
| `POST /trades/modify` | `#position-row-{instrument}` | `outerHTML` | Server returns the re-rendered `<tr>` reflecting the new state (e.g., updated trail stop, new contracts count, manual badge if `manual_stop` was set or removed). |
| `GET /trades/close-form?instrument=...` | `#position-row-{instrument}` | `outerHTML` | Server returns a 2-row block: original row above + confirmation panel row below (see §Destructive confirmation pattern) |
| `GET /trades/modify-form?instrument=...` | `#position-row-{instrument}` | `outerHTML` | Server returns a 2-row block: original row above + inline modify form row below |

**On 400 (validation failure)** for any POST: server returns the field-level error JSON `{"errors": [{"field": "...", "reason": "..."}]}` with status 400. HTMX does NOT swap (default behaviour: only 2xx triggers a swap). The form's `hx-on::after-request` handler parses the response body, populates the `.error` region above the form, and leaves the form values intact for correction. This handler is the only inline JS Phase 14 ships beyond the HTMX library itself.

**Confirmation banner out-of-band swap:** Successful 200 responses from open/close/modify additionally include an `HX-Trigger` header and an out-of-band `<div hx-swap-oob="innerHTML:#confirmation-banner">…</div>` so the confirmation banner renders without disturbing the row swap. Planner decides whether the banner slot is added as a fixed `<div id="confirmation-banner">` at the top of the Open Positions section or just below the page `<header>`.

### Decision 4 — Error display pattern

**Decision: inline `<div class="error">` block rendered above the form's submit button**, populated by HTMX `hx-on::after-request` handler from the 400 JSON body. No banner, no toast, no modal.

**Region structure (server-rendered as empty on every dashboard load; populated client-side on 400):**

```html
<div class="error" role="alert" aria-live="polite" hidden>
  <p class="error-heading">Could not save trade:</p>
  <div class="error-rows">
    <!-- one <div class="error-row"> per error, populated by hx-on::after-request -->
  </div>
</div>
```

**Visual contract:**
- `display: none` (via `hidden` attribute) when empty
- When populated: 1px solid `_COLOR_SHORT` left border (4px-wide accent), `_COLOR_SURFACE` background, `--space-3` padding, `--space-4` margin-top
- Heading line in `_COLOR_TEXT` weight 400 — matches body weight, not bolded, to avoid shouting
- Each error row in `_COLOR_TEXT_MUTED` weight 400, monospace `field` then colon-space then prose `reason` in regular sans (e.g., `<code>entry_price</code>: must be greater than 0`)
- Region clears (back to `hidden`) on the next form submission attempt — operator never has to manually dismiss

**The inline JS handler (the only client-side script Phase 14 ships beyond HTMX itself):**

```js
// hx-on::after-request handler — exact prose for executor to embed inline
function handleTradesError(evt) {
  if (evt.detail.successful) return;  // 2xx already swapped
  const errorBox = evt.target.closest('section').querySelector('.error');
  const status = evt.detail.xhr.status;
  if (status === 401) {
    errorBox.innerHTML = '<p class="error-heading">Auth header missing or wrong — refresh the page</p>';
  } else if (status === 400) {
    let body;
    try { body = JSON.parse(evt.detail.xhr.responseText); } catch (e) {
      errorBox.innerHTML = '<p class="error-heading">Server error — see journald</p>';
      errorBox.hidden = false;
      return;
    }
    const heading = '<p class="error-heading">Could not save trade:</p>';
    const rows = (body.errors || []).map(
      e => `<div class="error-row"><code>${e.field}</code>: ${e.reason}</div>`
    ).join('');
    errorBox.innerHTML = heading + '<div class="error-rows">' + rows + '</div>';
  } else if (status === 409) {
    // Conflict: surface the body verbatim (server-formatted prose)
    errorBox.innerHTML = '<p class="error-heading">' + evt.detail.xhr.responseText + '</p>';
  } else {
    errorBox.innerHTML = '<p class="error-heading">Server error — see journald: journalctl -u trading-signals-web</p>';
  }
  errorBox.hidden = false;
}
```

**XSS posture for the error region:** `e.reason` and the 409 body text come from server-controlled prose (Pydantic validators + Phase 14 D-01 conflict messages). They are NOT user-controllable. The inline handler uses `innerHTML` for layout speed; if any future REQ allows operator-supplied free-text in error reasons, the planner MUST switch this to `textContent` per CLAUDE.md §Global Patterns "innerHTML with dynamic data requires escaping". Phase 14 is safe under current scope.

### Decision 5 — Destructive confirmation pattern

**Decision: 2-stage HTMX inline confirmation in the same row** — clicking `Close` swaps the row to a confirmation panel that includes the exit-price input and `Confirm close` / `Cancel` buttons. NO `confirm()` JavaScript dialog (jarring, browser-styled, can't be themed).

**Stage 1 (operator clicks `Close` on a row):**

```
Row before:                                               Row after Close click:
┌────────────────────────────────────────────────────┐    ┌──────────────────────────────────────────────────────────────────────┐
│ SPI200  LONG  $7,800.50  $7,825.00  2  Lvl 0  …  │    │ Close LONG SPI 200 (2 contracts) at exit price [_____] [Cancel] [Confirm close] │
└────────────────────────────────────────────────────┘    └──────────────────────────────────────────────────────────────────────┘
```

The confirmation row spans all 9 columns via `<td colspan="9">` and contains:
1. Prose: `Close LONG SPI 200 (2 contracts) at exit price`
2. `<input type="number" step="0.01" min="0" name="exit_price" required autofocus>` — autofocused so the operator can type immediately
3. `Cancel` button (text-only, no colour) — `hx-get="/trades/cancel-row?instrument=SPI200"` reverts to the original row
4. `Confirm close` button (`_COLOR_SHORT` text + border) — submits the close POST. Initially `disabled`; enabled when `exit_price` input has a parseable positive number (HTMX `hx-trigger="keyup changed delay:100ms"` on the input toggles `disabled`)

**Stage 2 (operator clicks `Confirm close`):** standard `POST /trades/close` per §Decision 3.

**Why not a JavaScript `confirm()` dialog:**
- Browser-styled (can't be themed dark to match the dashboard)
- Modal blocks the whole page (overkill for a row-level action)
- No way to capture the `exit_price` input in the same dialog — would need a separate prompt or pre-confirm input field

**Why not submit-immediately:**
- Close is irreversible (mutates `state.account`, appends to `trade_log`). A typo on the row's Close button would be costly.

**Modify is NOT destructive** and uses the same 2-row inline-form pattern WITHOUT a `Confirm modify` step — Modify can be re-applied or reversed by a follow-up Modify, so the inline form's `Save` button submits directly.

### Decision 7 — Open form layout (required + optional fields)

**Decision: 4 required fields rendered as a single horizontal row of inline form fields; 4 optional fields hidden inside a collapsed `<details>` element labelled `Advanced`.**

## Accessibility Posture

Phase 14 ADDS form controls and inline buttons — accessibility is in scope where new surface ships. v1.0 dashboard semantics (already in place via existing `<section aria-labelledby>`, `<th scope="col">`, `<caption class="visually-hidden">`) are preserved unchanged.

**Required accessibility behaviours for new Phase 14 surfaces:**

- Every `<input>` and `<select>` MUST have an associated `<label for="...">` (not implicit wrapping; explicit `for`/`id` per CLAUDE.md §Frontend HTML shells without JavaScript). Planner generates IDs via `f'open-form-{field_name}'` pattern.
- The `Open Position` button MUST have `type="submit"` (default for `<button>` inside form, but explicit per the same CLAUDE.md guidance).
- The `Close` and `Modify` row buttons MUST have `type="button"` (NOT submit — they're not inside a form).
- The error region uses `role="alert" aria-live="polite"` so the screen reader announces validation failures without stealing focus.
- The `<details>` element is keyboard-accessible by default (Enter/Space toggles). No additional ARIA needed.
- The `Confirm close` button is `disabled` until `exit_price` is parseable positive — this MUST be a real `disabled` attribute (not just visual styling) so screen readers and keyboard nav skip it.
- Auto-focus on the `exit_price` input inside the close confirmation row is intentional — the operator's mental model is "I clicked Close, now type the exit price". Single-operator app; not a multi-step wizard pattern.
- The `manual` badge MUST include the `title=` attribute documented in §Decision 6. Title alone is not full a11y, but Phase 14 doesn't need to gold-plate this — operator is aware of the semantic from having set it themselves.

**Out of scope for Phase 14 a11y (deferred to future phases or explicit work):**
- Screen-reader-friendly announcements of confirmation banner content (the banner is a visual courtesy; the actual outcome is reflected in the table row swap which IS announced by `aria-live` on the tbody if the planner adds it — but Phase 14 doesn't require this).
- Tab order optimisation across the open form / table actions / advanced fields.
- WCAG colour-contrast audit beyond the existing v1.0 palette.

## Browser / Client Compatibility

- HTMX 1.9.x supports all modern browsers (last 2 versions of Chrome, Firefox, Safari, Edge). Same matrix as Chart.js 4.4.6 already loaded by v1.0.
- `<details>` / `<summary>` is universally supported in those browsers.
- `<input type="date">` / `<input type="number">` are universally supported but with vendor-styled controls — Phase 14 explicitly does NOT normalise their appearance via CSS (vendor styling is acceptable for a single-operator tool; cross-vendor pixel-perfect parity is gold-plating).
